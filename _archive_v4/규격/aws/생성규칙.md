# aws 플러그인 생성규칙

> buildflow cycle-2 ②명세. **런타임규격.md conform** — ActivitySpec 형태는 런타임규격 §2-1이 SoT.
> 대상: Claude(생성기). LearningProfile → `cloud-task` ActivitySpec 배열 생성 절차.
> 원칙: 사실 웹그라운딩 의무(AWS 서비스·CLI 정확성), 기계검증 가능 checks 작성.

---

## 0. 사전 조건: 웹그라운딩 의무

**cloud-task 생성 전 반드시 웹서치로 확인**:

1. **CLI 명령 정확성**: AWS CLI v2 현행 파라미터·플래그 (deprecated/변경 잦음).
2. **서비스별 리소스 식별자**: boto3 `head_bucket` / `describe_instances` / `get_function` 등 API가 실제로 받는 identifier 형식.
3. **리전 제약**: 서비스별 리전 가용성 (예: S3 us-east-1 LocationConstraint 생략 규칙).
4. **LocalStack 지원 여부**: Community 에디션 지원 서비스 목록 확인 (Pro 전용 서비스 과제 금지).

임베딩 기억으로 AWS CLI를 단정하지 말 것 — API/파라미터 변경 주기가 짧음.

## 1. LearningProfile 입력

깊이스무고개.md §STEP2에서 산출된 `LearningProfile` 수신:

```ts
LearningProfile = {
  subject: string,                    // 예: "AWS Cloud Practitioner"
  subject_structure: { ... },         // 웹조사 결과
  estimated_level: string,            // 예: "입문 — CLI 기초 경험 있음"
  goal: enum,                         // 시험 | 실무 | 교양
  scope: string[],                    // 예: ["S3","EC2","IAM","Lambda"]
  depth_ratio: { theory, practice },  // 예: { theory:0.3, practice:0.7 }
  volume: string,                     // 예: "30문항, 2주"
  target_plugins: ["aws"]
}
```

## 2. 활동 수량·분포 규칙

| 규칙 | 내용 |
|---|---|
| 수량 | `volume`에서 aws 배정 수 추산. 미명시 시 scope 토픽당 2~4문항 |
| 서비스 비중 | `scope` 비율에 따라 분산. 단일 서비스 집중 금지(최대 40%) |
| 난이도 분포 | estimated_level 기반: 입문=easy 60%/medium 40%, 중급=easy 30%/medium 50%/hard 20% |
| depth_ratio | practice > 0.6이면 `config-equals` check 포함 비중 높임(속성까지 검증) |
| goal=시험 | 자격증 시험 출제 토픽 우선 (웹그라운딩으로 시험 범위 확인) |

## 3. activity_id 명명 규칙

```
aws-{service}-{action}-{3자리 일련번호}
```

예: `aws-s3-create-bucket-001`, `aws-ec2-launch-instance-001`, `aws-lambda-create-function-001`

- 소문자·하이픈만. 정규식: `^aws-[a-z0-9]+-[a-z0-9-]+-\d{3}$`
- `plugin_id` const → `"aws"`, `type` const → `"cloud-task"`.

## 4. front 작성 규칙

### 4-1. prompt
- **학습 목적 명확화**: 왜 이 작업이 중요한지 1문장 포함.
- **구체적 리소스명 지정**: 채점 checks의 identifier와 일치. 모호한 "적절한 이름으로"는 금지.
- **리전 명시**: 필요 시 `ap-northeast-2` 등 구체적 리전 지정(LocalStack 기본 리전과 정합).
- 길이: 20~200자.

### 4-2. service
- boto3 client name과 일치: `s3`, `ec2`, `lambda`, `iam`, `dynamodb`, `sqs`, `sns`, `cloudwatch` 등.
- UI 표시용으로는 대문자(예: `S3`)를 쓰되, checks.service는 boto3 소문자 key 사용.

### 4-3. task
- 동사+목적어 형태, 20자 이내. 예: "S3 버킷 생성", "EC2 인스턴스 시작", "Lambda 함수 배포".

### 4-4. checkpoints
- 학습자가 순서대로 따라할 수 있는 절차.
- 최소 2개, 최대 6개.
- 각 항목: 동사 시작, 40자 이내.
- **채점 check와 대응**: 마지막 checkpoint는 반드시 결과 확인 단계 포함.

```
좋은 예:
  ["IAM 콘솔에서 역할 생성 클릭",
   "신뢰 정책에 Lambda 선택",
   "AdministratorAccess 권한 정책 연결",
   "역할 이름 'my-lambda-role'로 저장",
   "역할 목록에서 생성 확인"]

나쁜 예:
  ["적당히 역할 생성", "확인"]  // 모호, 리소스명 없음
```

### 4-5. cli_hint (선택)
- AWS CLI v2 현행 명령 (웹그라운딩 확인 필수).
- `--region ap-northeast-2` 포함 권장 (LocalStack 기본 리전).
- LocalStack 대상이면 `--endpoint-url http://localhost:4566` 추가 가능.
- 길이: 한 줄 ~ 5줄 이내.

## 5. grading.checks 작성 규칙

### 5-1. check 타입 선택 기준

| 과제 목표 | 권장 check 타입 |
|---|---|
| 리소스를 만들었는가 | `resource-exists` |
| 리소스의 특정 설정이 맞는가 | `config-equals` |
| 두 조건 모두 | `resource-exists` + `config-equals` 2개 |

### 5-2. resource-exists 작성 규칙

```jsonc
{
  "type": "resource-exists",
  "service": "s3",           // boto3 client name (소문자)
  "identifier": "my-bucket"  // 버킷명 / 인스턴스ID / 함수명 / 역할명 등
}
```

서비스별 identifier 형식 및 boto3 조회 API:

| service | identifier | boto3 조회 API |
|---|---|---|
| `s3` | 버킷명 (예: `my-bucket-2026`) | `head_bucket(Bucket=id)` |
| `ec2` | 인스턴스 ID (예: `i-0abc1234`) | `describe_instances(InstanceIds=[id])` |
| `lambda` | 함수명 (예: `my-function`) | `get_function(FunctionName=id)` |
| `iam` | 역할명 (예: `my-role`) | `get_role(RoleName=id)` |
| `dynamodb` | 테이블명 (예: `my-table`) | `describe_table(TableName=id)` |
| `sqs` | 큐 URL 또는 큐명 | `get_queue_url(QueueName=id)` |

**LocalStack Community 지원 서비스만 사용**: S3, EC2(기본), Lambda, IAM, DynamoDB, SQS, SNS, CloudWatch 등. EKS·ElastiCache 등 Pro 전용은 금지.

### 5-3. config-equals 작성 규칙

```jsonc
{
  "type": "config-equals",
  "service": "s3",
  "identifier": "my-bucket",
  "expected": { "versioning": "Enabled" }   // grade.py가 해석하는 속성 명세
}
```

`expected` 값은 grade.py가 boto3로 조회하는 실제 응답 필드와 대응해야 함.
웹그라운딩으로 API 응답 구조 확인 후 작성 (임베딩 단정 금지).

### 5-4. checks 배열 작성 원칙
- **최소 1개, 권장 1~3개**. 과도한 checks는 LocalStack 미지원 속성 포함 위험.
- 모든 check는 **LocalStack에서 기계검증 가능**해야 함.
- prompt에서 지정한 리소스명 = identifier — 반드시 일치.
- `config-equals`는 `resource-exists`보다 뒤에 배치 (존재 확인 후 속성 확인 순서).

## 6. back 작성 규칙

### 6-1. solution
- AWS CLI v2 전체 명령 (웹그라운딩 확인 필수).
- 복수 명령이면 순서대로 줄바꿈.
- 예시에 사용된 리소스명은 front.prompt의 리소스명과 동일.

### 6-2. explanation
- 핵심 개념 1~3문장.
- 자주 틀리는 함정(예: S3 리전 LocationConstraint 규칙, IAM 신뢰 정책 vs 권한 정책 구분).
- 음슴체(개조식): "~함", "~임", "~필요".

## 7. tags 작성 규칙

| 필드 | 내용 |
|---|---|
| `area` | AWS 서비스 범주. 예: `스토리지`, `컴퓨팅`, `네트워크`, `데이터베이스`, `보안/IAM`, `서버리스` |

| `unit` | 세부 작업. 예: `버킷 생성`, `인스턴스 시작`, `역할 생성`, `테이블 생성` |

## 9. 기계검증 체크리스트 (생성 후)

생성된 ActivitySpec 배열에 대해 아래 전부 pass:

| # | 항목 | 검증 방법 |
|---|---|---|
| G1 | 모든 ActivitySpec이 런타임규격 §2-1 JSON Schema validate 통과 | ajv 또는 jsonschema |
| G2 | `activity_id` 정규식 `^[a-z][a-z0-9-]{2,63}$` 전부 통과 | 정규식 검사 |
| G3 | `activity_id` 전체 배열에서 유일 | Set 중복 검사 |
| G4 | 모든 `front.prompt`의 리소스명이 대응 `checks[].identifier`와 일치 | 문자열 비교 |
| G5 | `config-equals` check에 `expected` 필드 존재 | 스키마 if/then |
| G6 | `checks[].service`가 boto3 소문자 client name | 허용목록 검사 |
| G7 | `checks[].service`가 LocalStack Community 지원 목록 내 | 허용목록 검사 |
| G8 | `grading.grader === "boto3"` | 스키마 const 검사 |
| G9 | `front.checkpoints.length >= 2` | 배열 길이 검사 |
| G10 | `back.solution` 내 CLI 명령에 `--endpoint-url` 또는 실제 AWS 지정 명시 | 텍스트 포함 검사 |
