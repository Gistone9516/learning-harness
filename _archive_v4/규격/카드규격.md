# 카드규격 (계약)

> buildflow ② 명세 산출물. 이 문서는 **기계검증 가능한 계약**이다(자유서술 아님).
> 상위 기획 = `../기획_v4_수렴.md`. 모순 시 기획_v4가 우선이며 이 문서는 [OPEN]으로 표시한다.
> 적용 경계(기획_v4 §2,§5): 자동채점 없음 · 합격확률 예측 없음 · 무의존 stdlib · 정적 HTML + localStorage.
> 채점은 입력 정규화 후 **문자열/키워드/빈칸 비교 + 자가O·X**까지만(시뮬레이션·실행채점 없음).
> **부분점수 없음**: 모든 type·mode는 이진 정/오답(`correct` | `incorrect`)만 산출한다. 세션내 재큐("다 맞출 때까지 다시 출제")가 교정장치다(기획_v4 §3).

---

## 0. 표기 규약 (이 계약의 메타규칙)

- 타입 표기: `string` `int` `float` `bool` `string[]` `object` `enum(...)` `null`.
- `필수`=항상 존재 / `옵션`=없을 수 있음(없으면 `기본값` 적용) / `조건부필수`=특정 type에서만 필수.
- 모든 enum 값은 **소문자 ASCII**, 허용값 외 입력은 빌드 에러(E-SCHEMA, §9).
- JSON 키는 **snake_case**. 트레일링 콤마 금지(stdlib `json` 직렬화 결과).
- 인코딩 UTF-8(BOM 없음). 줄바꿈 `\n`.
- **옵션 필드의 기본값 표기**: 표의 `기본값`은 "필드 부재 시 채점/처리에서 적용되는 의미값"이다. 작성자가 필드를 생략하면 그 의미로 동작한다. 카드 JSON에 명시적 `null`을 넣을 필요는 없다(부재 = 기본값). 구현은 부재와 명시적 기본값을 동일하게 처리해도 된다.

## 1. 카드 JSON 스키마

카드 1장 = JSON object 1개. deck = `Card[]` 배열을 네임스페이스로 감싼 구조(`../엔진규격.md` 소관, 여기선 카드 object만 정의).

### 1.1 공통 필드 (모든 type 공통)

| 필드 | 타입 | 필수성 | 기본값 | 허용값/제약 |
|------|------|--------|--------|-------------|
| `card_id` | string | 필수 | — | §6 규칙. 정규식 `^[a-z][a-z0-9-]{2,63}$` (소문자 시작, 소문자·숫자·하이픈, 3~64자). deck namespace 내 유일. |
| `schema_version` | int | 필수 | — | 현재 `1`. 증가 시 마이그레이션 shim 필요(엔진규격 소관). |
| `type` | enum | 필수 | — | `func` \| `proc` \| `recall_seq` \| `cloze` \| `judge` (§2) |
| `grade_mode` | enum | 필수 | — | `exact` \| `keyword` \| `cloze` \| `self` (§3). type별 허용조합 §2.6. |
| `front` | object | 필수 | — | 앞면(단서최소). §4.1 |
| `back` | object | 필수 | — | 뒷면(상세). §4.2 |
| `answer` | (type별) | 조건부필수 | — | type/grade_mode별 형태 §2·§3. `self`는 생략 가능. |
| `tags` | object | 옵션 | `{}` | §7 (빈출·배점 weight 등) |
| `links` | object | 옵션 | `{}` | 개념서 양방향 링크. §5 |
| `subject` | string | 필수 | — | 과목 네임스페이스 키(예: `comp1`). deck namespace와 일치(검증은 엔진규격 manifest 빌드 소관, §6.7 참조). |
| `unit` | string | 필수 | — | 단원/영역 id. 약점진단·진도바 집계 키. 비어있지 않음. |
| `enabled` | bool | 옵션 | `true` | `false`면 출제큐 제외(보류 카드). |

> `created`/`updated` 등 타임스탬프는 카드 데이터에 두지 않는다(콘텐츠는 git/소스가 진실). 학습 이력은 localStorage(엔진규격).

### 1.2 grading object (채점 파라미터, `answer`에 동반)

`grade_mode != self`인 경우 채점에 필요한 옵션은 `grading` object로 명시한다.

| 필드 | 타입 | 필수성 | 기본값 | 의미 |
|------|------|--------|--------|------|
| `accept` | string[] | 옵션 | `[]` | 추가 복수정답(전부 정답으로 인정). exact/cloze에서 사용. `answer` 단일값에 더해지는 보강용. 빈 배열·부재 = 추가정답 없음(`answer`만 정답). 비어있지 않은 원소만 유효(빈 문자열 원소는 무시). |
| `keywords` | object[] | 조건부필수 | — | keyword 모드 채점 단위. 최소 1개. §3.2 |
| `accept_map` | object | 옵션 | `{}` | cloze 빈칸별 추가정답. 키=빈칸번호(string), 값=string[]. §2.4 |
| `normalize` | string[] | 옵션 | 과목 config 기본 | 적용할 정규화 규칙 id 목록(§3.5). 카드별 오버라이드. **빈 배열 `[]` = 정규화 없음(원문 그대로 비교)**, **필드 부재 = 과목 config 기본 프로파일 적용**(둘은 다름). |
| `case_sensitive` | bool | 옵션 | `false` | true면 대소문자 구분(예: EXACT 함수 자체를 묻는 cloze). `lower` 규칙 비활성. |

> `normalize` 미지정(부재) 시 과목 config의 type별 기본 프로파일(`../생성규칙.md` config 스키마)을 사용. 카드에 두면 엔진은 카드값을 우선한다.
> `accept`는 옵션이다. `exact`/`cloze`에서 정답은 `answer`(단일값/빈칸맵)만으로 성립하며, `accept`/`accept_map`은 복수표기를 더하는 보강 필드다(§3.6, §8.8).

### 1.3 answer_spec — 생성물 (작성 대상 아님)

> **`answer_spec`은 generate.py가 빌드 시 평면 키에서 조립하는 런타임 객체다(SoT: _인터페이스계약 §13). 카드 콘텐츠 작성자는 `answer_spec`을 직접 작성하지 않는다.**

- **authoring SoT(평면 키)**: 카드 콘텐츠 md에서 작성하는 필드는 `answer`, `grading.accept`, `grading.keywords`, `grading.accept_map`(cloze 빈칸별 추가정답). 이 평면 키가 채점 데이터의 진실 원천이다.
- **generate.py 소유**: generate.py가 빌드 시 위 평면 키를 읽어 type별로 `answer_spec` 객체를 조립, deck Card에 삽입한다. 엔진은 `answer_spec`만 소비하며 런타임에 평면 키를 직접 참조하지 않는다.
- **type별 조립 결과** (generate.py가 생성하는 `answer_spec`의 필드 — 작성자 참고용):

  | type / grade_mode | answer_spec 산출 필드 | 원천 평면 키 |
  |---|---|---|
  | `func` / `judge` — exact | `accepted: string[]` | `answer` + `grading.accept` (OR 집합) |
  | `proc` — keyword | `requiredKeywords` | `grading.keywords` |
  | `recall_seq` — exact | `sequence: string[]` | `answer` 배열(순서 의미) |
  | `cloze` | `blanks` (0-base 키 맵) | `answer` 맵 + `grading.accept_map` |
  | `self` | 생략/빈값 | — |

- **검증**: 빌드 시 평면 키의 유효성(§8)을 검증한다. 빌드 결과물인 `answer_spec`의 스키마 검증은 `../엔진규격.md` 소관이다.

## 2. 5유형(type)별 형태

각 type은 `front`/`back`/`answer` 형태가 다르다. type ↔ grade_mode 허용조합은 §2.6.

### 2.1 `func` — 함수
- 목적: 함수 구문/인수 인출.
- `front.prompt`: 함수가 하는 일 또는 빈칸 시나리오(단서최소).
- `answer`: string (정답 함수식/함수명). 복수표기는 `grading.accept`.
- 예시 형태:
```json
{ "type":"func","grade_mode":"exact",
  "front":{"prompt":"조건에 맞는 셀의 합계를 구하는 함수는?"},
  "back":{"detail":"SUMIF(range, criteria, [sum_range])","note":"다중조건은 SUMIFS"},
  "answer":"SUMIF",
  "grading":{"accept":["SUMIF()"]} }
```
> `answer`만으로도 정답 성립(`accept` 없이 가능). 위 예시의 `accept`는 `()` 표기 변형을 더한 보강이다.

### 2.2 `proc` — 절차
- 목적: 메뉴경로/단축키/작업단계 인출.
- `front.prompt`: 달성 목표(예: "조건부 서식으로 상위10% 강조").
- `answer`: string (모범 절차 서술). 채점은 보통 `keyword`(핵심키워드 포함 여부).
- `back.detail`: 단계/메뉴경로/단축키/주의(개념서 절차 모델과 정합).

### 2.3 `recall_seq` — 순서재현
- 목적: 정해진 단계의 **순서** 인출.
- `answer`: string[] (순서가 의미 있는 단계 배열).
- `front.prompt`: 재현 대상.
- 채점: `exact`는 순서 포함 완전일치(정규화 후 step별 비교), `self`는 자가확인. **부분점수 없음**: step 일부만 일치하면 `incorrect`(앞 k개 일치 인정 안 함). 다 맞출 때까지 세션내 재큐.
- 형태:
```json
{ "type":"recall_seq","grade_mode":"exact",
  "front":{"prompt":"피벗테이블 만들기 클릭 순서?"},
  "answer":["데이터 선택","삽입 탭","피벗테이블","위치 지정","확인"],
  "grading":{"normalize":["trim","collapse_space"]} }
```

### 2.4 `cloze` — 빈칸
- 목적: 문장/수식 내 빈칸 인출.
- `front.text`: 빈칸 마커 포함 문자열. 마커 = `{{0}}`, `{{1}}` … (**0부터** 연속, 중복 출현 금지 — 같은 번호 마커는 정확히 1회). **빈칸 인덱스는 0-base**(엔진 matched/missed, UI data-blank-index, 생성 idx 전부 0-base 일치).
- `answer`: object — 키=빈칸번호(string, 0-base), 값=정답 string. 빈칸별 복수정답은 `grading.accept_map`.
- 형태:
```json
{ "type":"cloze","grade_mode":"cloze",
  "front":{"text":"=VLOOKUP(찾을값, 범위, {{0}}, {{1}})"},
  "answer":{"0":"열번호","1":"FALSE"},
  "grading":{"accept_map":{"1":["FALSE","0"]}} }
```
- `accept_map` (object, 옵션): 키=빈칸번호, 값=string[] 추가정답. 미지정 빈칸은 `answer`값만 정답. 특정 빈칸 키에 빈 배열 `[]`을 주는 것은 "추가정답 없음"과 동일(=미지정과 같음).
- **빈칸 입력 규칙**: 모든 빈칸을 채워야 한다. 사용자가 어느 빈칸을 비우면(빈 문자열) 그 빈칸은 오답으로 처리하며, 전 빈칸 정답이 아니므로 카드는 `incorrect`(§3.3). 정규화는 빈 문자열을 빈 문자열 그대로 둔다(정답이 빈 문자열이 아닌 한 불일치).

### 2.5 `judge` — 판단형(시나리오)
- 목적: 조건 시나리오 → 적절한 절차/함수 **선택**.
- `front.scenario`: 조건 서술.
- `front.options` (string[], 옵션): 선택지(있으면 객관식). 없으면 단답 입력(자유입력).
- `answer`: string (정답 선택지 텍스트 또는 정답 키). options 있으면 options 중 하나와 일치해야 함(빌드검증 E-OPTANS).
- **options 없는 단답 judge**: `exact` mode로 채점. `answer`(필요 시 `grading.accept` 보강)와 정규화 후 완전일치 비교. 오타 허용은 정규화 규칙(§3.5)으로만 흡수하며, 그 외 비일치는 `incorrect`(부분점수 없음).
- 형태:
```json
{ "type":"judge","grade_mode":"exact",
  "front":{"scenario":"VLOOKUP이 #N/A 발생, 정확일치 검색인데 못 찾음. 우선 점검할 것은?",
           "options":["range_lookup=FALSE 여부","셀 서식","시트 보호"]},
  "answer":"range_lookup=FALSE 여부" }
```

### 2.6 type × grade_mode 허용 매트릭스

| type \ mode | exact | keyword | cloze | self |
|-------------|:-:|:-:|:-:|:-:|
| `func`        | O | △ | X | O |
| `proc`        | X | O | X | O |
| `recall_seq`  | O | △ | X | O |
| `cloze`       | X | X | O | O |
| `judge`       | O | X | X | O |

- O=권장/정상, △=허용하나 비권장, X=빌드 에러(E-MODE).
- `cloze` type은 `cloze` mode 또는 `self` mode만 허용한다. `cloze` mode는 `cloze` type에서만 쓴다(다른 type에서 `cloze` mode = E-MODE).
- 모든 type은 `self` 사용 가능(자가O·X 폴백). 따라서 `cloze` type도 `self` mode 허용(자동비교 대신 자가확인).

## 3. 4채점모드(grade_mode)와 정규화 규칙

채점 알고리즘 = **정규화(normalize) → 비교(compare)**. 정규화는 입력(사용자답)과 정답 양쪽에 동일 적용. 무의존 stdlib·결정적(같은 입력→같은 결과). 모든 모드는 이진 정/오답을 산출(부분점수 없음). 채점기/정규화/cloze파서는 테스트셋 동반 필수(기획_v4 §5, 검증게이트 ④).

### 3.1 `exact` — 구문 정확일치
- 절차: 정규화 후 사용자답이 `answer` 또는 `grading.accept[]` 중 **하나와 완전일치**하면 `correct`, 아니면 `incorrect`.
- `recall_seq`: step 배열을 step별 정규화 후 **순서 포함 동일** 시 `correct`, 아니면 `incorrect`(부분일치 없음).

### 3.2 `keyword` — 핵심키워드 포함
- `grading.keywords`: object[](최소 1개). 각 원소:
  | 필드 | 타입 | 필수 | 기본 | 의미 |
  |------|------|------|------|------|
  | `any` | string[] | 필수 | — | 이 그룹 중 **하나라도** 포함되면 그룹 충족(동의어 그룹). 최소 1개 원소. |
- 정답 판정: **모든 키워드 그룹이 충족되면 `correct`, 하나라도 미충족이면 `incorrect`**(부분점수 없음 — N그룹 중 일부 충족도 `incorrect`). 포함 검사는 정규화 후 부분문자열 매칭.
- v1에는 그룹 단위 `required`/가점 개념이 없다. 모든 그룹이 정답 필수 그룹이다(가점 기능 제거 — 이진 판정만).

### 3.3 `cloze` — 빈칸 비교
- 빈칸별로 사용자답을 정규화 후 `answer[n]` 또는 `accept_map[n][]`과 비교.
- **전 빈칸 정답 시 `correct`, 하나라도 오답이면 `incorrect`**(부분점수 없음). 빈칸 단위 정/오 표시(피드백)는 UI 소관(UI규격)이며, 카드 판정 자체는 이진이다.

### 3.4 `self` — 자가 O·X
- 채점 자동판정 없음. 정답 제시 후 사용자가 직접 O/X 입력 → 그 값이 정/오답(`correct`/`incorrect`).
- `answer` 생략 가능(`back.detail`을 정답으로 제시). 정규화/비교 미적용.
- `self` 카드에 `answer`/`grading`이 존재해도 **무시**한다(스키마상 허용되나 채점에 미사용). 빌드는 경고 W-SELFEXTRA를 낼 수 있으나 실패는 아니다(§9).

### 3.5 정규화 규칙 카탈로그 (정규화 id → 동작)

각 id는 결정적 변환 함수. `grading.normalize[]` 또는 config 기본 프로파일에서 **지정된 순서대로** 적용한다(적용 순서는 `normalize[]` 배열 순서가 곧 계약이다 — 아래 표는 동작 정의이며 순서를 강제하지 않는다).

| id | 동작 |
|----|------|
| `trim` | 앞뒤 공백 제거. "공백" = 스페이스(U+0020)·탭(U+0009)·개행(U+000A,U+000D)·NBSP(U+00A0)·전각공백(U+3000). |
| `collapse_space` | 위 "공백" 정의의 연속 공백 1개 이상 → 단일 스페이스(U+0020) 1개. 문자열 양끝 공백은 trim과 별개로 남을 수 있으니 보통 `trim`과 함께 쓴다. |
| `strip_all_space` | 위 "공백" 정의의 모든 공백 문자를 제거. |
| `lower` | 대문자→소문자(ASCII A–Z → a–z만 변환, 비ASCII 미변경). `case_sensitive:true`면 비활성. |
| `fullwidth_to_halfwidth` | 전각 영숫자(U+FF10–FF19, U+FF21–FF3A, U+FF41–FF5A)·전각 ASCII 구두점/기호(U+FF01–FF5E 영역의 가시 ASCII 대응 문자)·전각 통화기호 등을 대응 반각 ASCII로 변환. 대응표가 정의된 문자만 변환, 그 외는 미변경. (예: `（）＝，＄Ａ` → `()=,$A`) |
| `unify_cell_dollar` | 셀참조의 절대/혼합참조 `$`를 제거하여 상대참조 형태로 통일. 셀참조 = 정규식 `\$?[A-Za-z]{1,3}\$?[0-9]{1,7}` 매치 토큰. 매치 내 모든 `$` 제거(예: `$A$1`,`A$1`,`$A1` → `A1`; 이후 `lower` 적용 시 `a1`). 비셀참조의 `$`는 미변경. |
| `unify_arg_sep` | 함수 인수구분자 통일 — 구분자로 쓰인 세미콜론(`;`, U+003B)·전각세미콜론(`；`, U+FF1B)·전각콤마(`，`, U+FF0C)를 반각콤마 `,`(U+002C)로 치환. 닫힌 집합 = {`;`, `；`, `，`}. 치환 후 콤마 앞뒤 공백 제거. |
| `strip_trailing_paren` | 토큰 끝의 빈 괄호쌍 `()`(여닫이 사이 공백 없음)을 제거하여 함수명만 물을 때 `SUMIF()`↔`SUMIF`를 동일시. |
| `nfkc` | 유니코드 NFKC 정규화(stdlib `unicodedata.normalize`). |

규칙 상호작용:
- `strip_all_space`와 `collapse_space`는 상호배타(둘 다 지정 시 `strip_all_space` 우선 적용, 경고 W-NORM — 빌드 검증 §8.10).
- `case_sensitive:true`면 `lower`가 목록에 있어도 비활성(예: 함수명 EXACT 자체를 답으로 묻는 경우).
- `nfkc`와 `fullwidth_to_halfwidth`는 전각→반각 영역이 겹친다. 둘 다 지정해도 **결과는 멱등**이다(NFKC가 먼저 반각화하면 `fullwidth_to_halfwidth`는 변환 대상이 없어 no-op, 역순도 동일 수렴). 충돌 없음.
- **전역 synonyms 치환 위치(계약, cross-contract 고정)**: 과목 config의 전역 동의어 사전(§3.6) 치환은 `lower` 적용 **직후, 비교 직전**에 수행한다. 즉 정규화 파이프라인 = [지정된 normalize 규칙들] → [synonyms 표제어 치환] → [비교]. 이 위치는 카드규격·생성규칙·엔진규격이 동일하게 따른다.

### 3.6 동의어 / 복수정답
- **복수정답**: `grading.accept[]`(전체답 보강), `grading.accept_map[n][]`(빈칸별 보강). OR 비교(`answer` 또는 이들 중 하나라도 일치=정답).
- **동의어**(keyword): `keywords[].any[]` 그룹 내 OR.
- **전역 동의어 사전**: 과목 config의 `synonyms`(`../생성규칙.md` config 스키마) — 정규화 파이프라인의 `lower` 직후·비교 직전(§3.5)에 표제어로 치환. 카드 `accept`는 사전 적용 후에도 추가 보강용.

## 4. 앞면(front)·뒷면(back) 분리

원칙(기획_v4 §3): 앞=**단서최소 인출**(읽기자료화 방지), 뒤=**상세**. 앞면에 정답을 노출하는 필드 금지(빌드검증 E-LEAK).

### 4.1 front object (type별 필수 필드)

| type | 필수 필드 | 타입 | 제약 |
|------|-----------|------|------|
| `func` | `prompt` | string | 정답 함수명/식 포함 금지(E-LEAK) |
| `proc` | `prompt` | string | 목표만. 정답 절차 포함 금지 |
| `recall_seq` | `prompt` | string | 재현 대상만 |
| `cloze` | `text` | string | `{{n}}` 마커 1개 이상, **0..N-1 연속(0-base)**·중복 없음. answer 키와 마커번호 일치(E-CLOZE) |
| `judge` | `scenario` | string | — |

공통 옵션 front 필드:
| 필드 | 타입 | 기본 | 의미 |
|------|------|------|------|
| `hint` | string | 부재 | 선택적 추가단서(요청 시 노출). 정답 직접노출 금지(E-LEAK 검사 대상, §8.5). |

### 4.2 back object

| 필드 | 타입 | 필수성 | 의미 |
|------|------|--------|------|
| `detail` | string | 필수 | 상세 해설(함수 구문·인수·예제 / 절차 단계·메뉴·단축키). |

- `self` 모드는 `back.detail`을 정답 기준으로 제시한다(§3.4).

## 5. 개념서 링크 (links object)

> **카드 `links` = `{concept_ref?}` 만**(SoT: _인터페이스계약 §14). `related_cards`(concept→card 방향)는 **개념서 ConceptSection 필드**이며 카드 필드가 아니다(§12).

| 필드 | 타입 | 기본 | 의미 |
|------|------|------|------|
| `concept_ref` | string | 부재 | 연결 개념서 섹션 id(`../개념서규격.md` 섹션 id 규약과 일치). |

- 개념서→카드 역링크는 개념서규격이 `concept_ref`를 역색인하여 생성(여기선 카드측 단방향 선언만 계약).
- **참조 검증 빌드 의존성**:
  - `concept_ref`: 개념서가 함께 빌드되는 경우에만 교차검증한다. 개념서가 아직 빌드 대상에 없으면 검증을 **보류**(스킵)하고 경고 W-REFDEFER를 낸다(빌드 실패 아님). 개념서 포함 빌드에서 미존재 = E-REF(단, 이 E-REF는 **개념서 ConceptSection 소관** — 카드 빌드 단독으로는 E-REF를 발생시키지 않으며 W-REFDEFER만 냄, SoT §12).

## 6. card_id 안정화 규칙

기획_v4 §5: card_id = **작성자 명시 안정 id**(content-hash 아님). 오탈자 수정에도 오답이력 보존이 목적.

규칙(계약):
1. 형식: 소문자 시작, 소문자·숫자·하이픈, 3~64자. 정규식 `^[a-z][a-z0-9-]{2,63}$`. **대문자·언더스코어·콜론 금지**(대소문자 FS 충돌·localStorage 키 콜론 분리자 회피). 하이픈 **세그먼트 강제 안 함**(있어도 됨).
2. 권장 구조: `<subject>-<unit>-<type약칭>-<연번|슬러그>` 예: `comp1-ss-func-0007`. **권장 구조와 type약칭 코드표는 명명관습(스타일가이드)일 뿐 기계검증 대상이 아니다**(검증은 정규식+유일성만). 본 계약은 약칭표를 강제하지 않는다.
3. **불변(immutable)**: 한 번 배포된 card_id는 변경 금지. 텍스트/정답 수정은 같은 id 유지(이력 보존).
4. 유일성: deck(subject 네임스페이스) 내 유일. 전역 유일성은 deck namespace 분리로 보장한다(card_id 문자열 자체에 subject 접두 포함을 강제하지는 않음 — 유일성은 namespace+card_id 조합으로 성립).
5. 삭제: id 재사용 금지(과거 이력과 충돌). 폐기 카드는 `enabled:false` 권장, 영구삭제는 id를 영구 retire.
6. 생성기는 card_id를 **부여하지 않고 검증만** 한다(작성자 명시). 누락 시 E-IDMISSING, 정규식 위반 시 E-IDFORMAT, deck 내 중복 시 E-IDDUP.
7. `subject` 필드 ↔ deck namespace 일치 검증은 deck 빌드/manifest 단계(`../엔진규격.md`·`../생성규칙.md` 소관)에서 수행한다. 본 카드 계약은 `subject`가 비어있지 않은 string임만 검증한다(E-SCHEMA).

## 7. 태그(tags) — 빈출·배점 weight

출제 우선순위(기획_v4 §3): Leitner due-first → 그 안에서 **빈출·배점 weight 정렬**, 신규카드는 weight 우선.

> **`freq` 필드 폐기(SoT §6)**: 빈출·배점 신호는 `weight` 하나로 일원화. `freq`는 이 규격에 존재하지 않는다.

| 필드 | 타입 | 필수성 | 기본값 | 허용값/제약 |
|------|------|--------|--------|-------------|
| `weight` | float | 옵션 | `5` | 범위 `[1, 10]`. 출제 정렬 가중치(배점·중요도 종합). 클수록 우선. 미지정 시 기본 5. 작성자가 빈출·중요 카드에 직접 높은 값 부여. |
| `area` | enum | 옵션 | 부재 | `written` \| `practical` (필기/실기). 합격경로 진단 집계 키. |
| `subarea` | enum | 옵션 | 부재 | area별 허용집합: `computer` \| `spreadsheet` \| `database`(area=`written`) · `excel` \| `access`(area=`practical`). 진단 직교축. 점결합 표기(`written.computer`) 금지 — 항상 {area, subarea} 2필드. |

- **weight 입력 계약**: `weight`는 `float`, 범위 `[1, 10]`. 작성자가 빈출·중요 카드에 직접 높은 값을 부여한다. 카드에 `weight`가 있으면 그 값이 권위값이다(엔진은 그대로 사용). **미지정 시 기본 5**. **freq 파생식은 폐기**(dangling 제거) — `weight`는 `freq`에서 파생하지 않는다.
- 엔진은 범위 밖 입력을 `[1, 10]`으로 클램프, 비숫자 입력은 기본 5로 폴백(런타임 무중단). 정렬식은 `../엔진규격.md` 소관(카드규격은 입력값·범위·기본값만 계약).

## 8. 카드 검증 규칙 (생성기 빌드타임, 무의존 stdlib)

생성기(`../생성규칙.md`)는 deck 빌드 시 카드마다 아래를 검증. error 하나라도 위반 시 빌드 실패(해당 카드 id·규칙코드 보고). warning은 빌드 통과하되 보고.

1. 스키마: 필수필드 존재, 타입 일치, enum 허용값(§1,§2,§3). `subject` 비어있지 않음.
2. type×mode 매트릭스 위반 없음(§2.6). `cloze` mode는 `cloze` type에서만, `cloze` type은 `cloze`/`self` mode만.
3. cloze: `front.text`의 `{{n}}` 마커가 **0..N-1 연속(0-base)**·중복 없음, `answer` 키 집합과 정확히 일치(§2.4). 위반 E-CLOZE.
4. judge: `options` 존재 시 `answer`가 options 중 하나(정규화 후 비교). 위반 E-OPTANS.
5. front 정답누출 검사(E-LEAK, best-effort): func/proc/recall_seq의 `front.prompt` **및 옵션 `front.hint`**에 정답 문자열(정규화 후)이 그대로 포함되지 않음. `front.media`(경로)는 검사 제외.
6. card_id: 정규식 통과(E-IDFORMAT) + 누락 아님(E-IDMISSING) + deck 내 유일(E-IDDUP)(§6).
7. links: `concept_ref`는 개념서 동반 빌드 시 교차검증, 아니면 보류 경고 W-REFDEFER(§5). `related_cards` dangling 검증은 **개념서 ConceptSection 소관** — 카드 빌드는 E-REF를 발생시키지 않음(SoT §12).
8. grading 필수 필드: `keyword` 모드 → `grading.keywords` 존재·최소 1그룹(각 그룹 `any` 최소 1원소). `cloze` 모드 → `answer` object가 마커와 정합(§8.3). `exact` 모드 → `answer` 존재(단일 string, recall_seq는 string[]). **`accept`/`accept_map`은 필수가 아니다**(보강 옵션).
9. tags 범위: `weight∈[1,10]`(존재 시; 범위 밖은 엔진이 클램프). `area∈{written,practical}`, `subarea`는 area별 허용집합(§7). **`freq` 필드는 폐기됨 — 존재 시 E-SCHEMA**(허용값 없는 알 수 없는 필드 처리와 동일).
10. 정규화 규칙 상호배타: `grading.normalize`에 `strip_all_space`와 `collapse_space` 동시 존재 시 경고 W-NORM(빌드 통과, §3.5).
11. 정규화 id 유효성: `grading.normalize`의 모든 id가 §3.5 카탈로그에 존재. 미존재 id = 빌드 실패 E-SCHEMA(§9). (전역 synonyms 사전은 정규화 id가 아니라 별도 치환 단계이므로 이 검사 대상 아님 — config 주입.)
12. self 부가필드: `self` 모드 카드에 `answer`/`grading` 존재 시 경고 W-SELFEXTRA(무시되는 필드 통지, 빌드 통과).

## 9. 에러/경고 규약 (코드 · 의미)

| 코드 | 수준 | 의미 |
|------|------|------|
| `E-SCHEMA` | error | 필수필드 누락/타입불일치/enum 위반/알 수 없는 normalize id |
| `E-MODE` | error | type×grade_mode 매트릭스 위반(§2.6) |
| `E-CLOZE` | error | cloze 마커↔answer 키 불일치/마커 비연속/마커 중복 |
| `E-IDMISSING` | error | card_id 누락 |
| `E-IDFORMAT` | error | card_id 정규식 위반 |
| `E-IDDUP` | error | deck 내 card_id 중복 |
| `E-REF` | error | links 참조 대상 미존재(동반 빌드 범위 내) — **카드 빌드에서는 미발생**; concept_ref 미검증 시 W-REFDEFER. related_cards dangling 검증은 개념서 ConceptSection 소관(SoT §12) |
| `E-LEAK` | error | front(prompt/hint)에 정답 노출(best-effort) |
| `E-OPTANS` | error | judge answer가 options에 없음 |
| `W-NORM` | warning | 상호배타 정규화 규칙 동시지정(strip_all_space + collapse_space) |
| `W-REFDEFER` | warning | concept_ref 교차검증 보류(개념서 미동반 빌드) |
| `W-SELFEXTRA` | warning | self 모드 카드에 무시되는 answer/grading 존재 |

- 채점 런타임 에러규약(입력 빈값·타입불일치·알 수 없는 normalize id 등)은 `../엔진규격.md`와 공유한다. 본 계약 정의: 빌드타임에 알 수 없는 normalize id = E-SCHEMA(빌드 실패). 런타임 입력 타입 검증(예: cloze에 string, self에 'o'/'x' 외 값)은 엔진규격 §3.2 `ScoreInputError` 소관이다(본 계약은 데이터 형태만, 런타임 가드는 엔진).

## 10. 인터페이스 시그니처 (채점 코어가 따라야 할 계약)

엔진 채점 코어(`엔진/`, 범용)는 아래 시그니처를 만족해야 한다(과목특화는 config 주입, 코어 무오염 — 기획_v4 §5). 언어중립 의사시그니처(JS/Python 공통 의미). **반환형의 상세 구조는 `../엔진규격.md`가 권위**이며, 여기서는 카드 데이터가 채점에 어떻게 들어가는지의 계약 접점만 명시한다.

```
normalize(text: string, rules: string[], opts: {case_sensitive: bool}) -> string
  // §3.5 카탈로그를 rules 순서대로 적용. 결정적. 알 수 없는 rule id는 빌드타임에 차단됨(E-SCHEMA) 전제.

grade(card: Card, user_answer: any) -> { correct: bool }
  // user_answer 타입은 type별:
  //   func/proc/judge = string
  //   recall_seq      = string[]
  //   cloze           = string[]  // 0-base 위치 배열. blanks[i] = i번 빈칸 사용자 입력(SoT §13·§14).
  //                                // UI가 {n:string} 맵을 마커 순서(0,1,…)대로 string[]로 변환해 전달.
  //   self            = enum('correct'|'incorrect')
  //                     // UI 레이어가 o→correct / x→incorrect 로 변환 후 전달(변환 주체=UI, SoT §5).
  //                     // 엔진 grade()는 'correct'|'incorrect'만 수용; 'o'/'x' 직접 전달 시 ScoreInputError.
  // 부분점수 없음: 반환은 이진 correct만 의미상 계약. (피드백·빈칸별 표시 등 부가 반환 구조는 엔진규격 ScoreResult 소관)
  // self 모드: correct = (user_answer == 'correct'). 자동비교 없음.
  // 잘못된 user_answer 타입의 런타임 처리(throw 등)는 엔진규격 §3.2 소관.

parse_cloze(text: string) -> { template: string, blanks: int[] }
  // {{n}} 마커 추출. 비연속/중복 시 E-CLOZE.
```

- `grade`는 부수효과 없음(localStorage 갱신은 호출측). 같은 (card, user_answer)→같은 결과.
- 과목별 정규화 옵션·동의어·모드매핑은 config로 주입되어 `normalize`/`grade`의 인자(`rules`, `synonyms`)로 전달(엔진 코드 분기 없음). 전역 synonyms는 `lower` 직후·비교 직전 치환(§3.5).

## 11. 검증 게이트 연동 (기획_v4 §5 ④)

- 본 계약의 §3(채점), §3.5(정규화), §2.4/§10(cloze 파서)은 **테스트셋 동반 필수**. 테스트셋은 (입력, 정규화기대, 정/오답기대) 케이스 표. 위치/형식은 `../엔진규격.md`·검증 게이트 산출과 합의.
- 최소 케이스(필수 커버): 대소문자 / 앞뒤·내부공백(개행·NBSP·전각공백 포함) / 셀참조 `$` 4형태 / 전각반각(`＝（），＄`) / 인수구분자(`;`·`；`·`，` → `,`) / 함수명 `()` 유무 / 복수정답 OR(`answer`만 vs `accept` 보강) / cloze 빈칸 일부오답·빈칸 비움 / keyword 그룹 일부충족(=incorrect) / recall_seq 부분일치(=incorrect) / self O·X / synonyms 치환(`lower` 직후 위치).
