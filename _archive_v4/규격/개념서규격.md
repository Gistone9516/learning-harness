# 개념서규격 (계약 v1)

> buildflow ② 명세. 이 문서는 **기계검증 가능한 계약**이다. 개념서 콘텐츠(.md 소스)와 generate.py 빌드 산출(개념서 JSON/HTML), 그리고 개념서↔카드 양방향 링크의 필드·타입·허용값·에러규약을 정의한다.
> 경계(기획_v4 준수): 자동채점 없음 · 합격확률예측 없음 · 무의존(stdlib) · 정적 HTML + localStorage. 개념서는 **읽기 레퍼런스**이며 채점/풀이 로직을 포함하지 않는다.
> 의존 계약(SoT 분담):
> - `card_id` 형식·카드측 역링크 필드(`links.concept_ref`)는 **카드규격.md**가 소유한다. 본 문서는 그 형식을 **참조만** 한다(중복정의 금지). deck/namespace는 card_id가 인코딩하지 않으며(_인터페이스계약 §2), 카드의 `subject` 필드 + manifest에서 온다 — "card_id에서 deck 추출" 규칙은 존재하지 않는다.
> - `manifest`/deck 전역 등록 변수명·`concepts` 등록 네임스페이스 키는 **생성규칙.md / 엔진규격.md**가 소유한다. 본 문서는 `concepts` 배열 **원소 스키마**만 계약한다.
> - 충돌 시 해당 SoT 우선. `[OPEN]` 표시는 추측 금지 항목(의도/스코프 함의로 본 문서가 확정 불가).

## 0. 용어·식별자 형식 (machine-checkable)

| 토큰 | 정규식(ECMA/PCRE) | 비고 |
|------|------|------|
| `section_id` | `^[a-z]{2,8}\.[a-z0-9_]+(\.[a-z0-9_]+)*$` | 네임스페이스 prefix + 점 계층. 예 `comp.calc.sumifs`, `comp.basic.cellformat` |
| `card_id` | 카드규격.md 소유 — 본 문서는 참조만 | 작성자 명시 안정 id(content-hash 아님). 정규식 SoT = `^[a-z][a-z0-9-]{2,63}$`(_인터페이스계약 §1: 소문자 시작, 소문자·숫자·하이픈, 3~64자. 대문자·언더스코어·콜론 금지). 하이픈 세그먼트 강제 안 함 |
| `subject_ns` | `^[a-z]{2,8}$` | 과목 네임스페이스. 컴활1급 = `comp` (확정, §6.4) |
| `func_name` | `^[A-Z][A-Z0-9.]*$` | Excel 함수명 대문자. 점 허용(`RANK.EQ`, `MODE.SNGL`). **id가 아님** — §0 소문자 정규화 규칙 비적용(콘텐츠 표기값) |
| `anchor` | `^[a-z0-9][a-z0-9-]*$` | HTML 내 섹션 앵커. 빌더가 `section_id`에서 **결정적으로 파생**(파생 알고리즘은 빌더 구현 자유, §3.3 충돌검사만 계약) |

- `section_id`·`subject_ns`·`anchor`는 **소문자 정규화 후** 비교(대소문자 무시 일치 금지 — 정규화로 단일화). `func_name`은 콘텐츠 표기값이므로 이 규칙에서 제외한다.
- id 중복은 빌드 에러(§5 `E_DUP_ID`). 전 과목 deck 통합 시 `subject_ns` prefix로 충돌 회피.

## 1. 개념서 콘텐츠 소스 = Markdown + Front-matter (작성 계약)

개념서 1개 **섹션** = `.md` 파일 1개(파일 1개 = 섹션 1개). 각 섹션은 YAML front-matter 필수.

- 섹션 단위 = **파일 1개**로 확정한다(§6.1). generate.py 파서는 콘텐츠 폴더의 `*.md` 파일 1개를 섹션 1개로 매핑한다. (파일 내 다중 H2 분할은 v1 비채택 — 파서·앵커·링크 단순화.)

### 1.1 섹션 front-matter 필드

| 필드 | 타입 | 필수 | 허용값/제약 | 기본값 |
|------|------|:---:|------|------|
| `section_id` | string | ✅ | §0 정규식. 과목 내 유일 | — |
| `subject` | string(enum) | ✅ | `subject_ns` 중 하나. 현재 허용집합 = {`comp`} | — |
| `area` | string(enum) | ✅ | `written` \| `practical` (필기/실기). _인터페이스계약 §11. `subarea`와 정합(§1.4-A) | — |
| `subarea` | string(enum) | ✅ | 아래 §1.4 영역 enum. `area=written` ⇒ `computer`\|`spreadsheet`\|`database`, `area=practical` ⇒ `excel`\|`access` | — |
| `unit` | string | ✅ | 단원 표시명(자유 텍스트, ≤40자) | — |
| `title` | string | ✅ | 섹션 제목(≤60자) | — |
| `ref_type` | string(enum) | ✅ | `function` \| `procedure` \| `concept` | — |
| `weight` | float | ✅ | [1, 10] (빈출·배점 가중; _인터페이스계약 §6. 자료조사 ★개수 매핑) | — |
| `related_cards` | string[] | ✅(빈배열 허용) | 각 원소 = `card_id`(카드규격 형식). 중복불가(§4 `E_DUP_LINK`). 개념→카드 방향 필드명(_인터페이스계약 §12) | `[]` |
| `prereq` | string[] | ❌ | 각 원소 = 선수 `section_id`. 존재해야 함(§4 `E_DANGLING_PREREQ`), 사이클 금지(§5 `E_CYCLE`) | `[]` |
| `tags` | string[] | ❌ | 각 `^[a-z0-9_]+$`, ≤8개. 위반 = `W_TAG_FORMAT`(경고) | `[]` |
| `source_ids` | string[] | ❌ | 각 원소 = string(자료조사 출처 식별자, 형식 비강제·추적용). 빈문자 금지 | `[]` |
| `schema_version` | int | ✅ | 리터럴 `1` 고정. 그 외 값 = `E_SCHEMA_VERSION`(거부, 마이그레이션 없음) | `1` |

- `related_cards`는 **개념서→카드 방향**의 정방향 선언이다(_인터페이스계약 §12 필드명). 역방향(카드→섹션)은 카드규격의 `links.concept_ref` 필드가 소유하며, 빌드시 **양방향 일치 검증**(§2.2 R-LINK-3).
- `weight`는 작성자가 **수동 명시**한다(필수, 자동매핑 아님 §6.2). 범위 = float [1, 10](_인터페이스계약 §6). 자료조사 ★ 매핑 가이드: ★1=2 … ★5=10(선형) 또는 작성자 판단. 별 표기 없는 절차/개념 섹션도 작성자가 [1,10]로 명시. 누락 시 `E_SCHEMA`(개념서는 필수 강제 — 미지정 시 카드규격의 기본 5 적용은 개념서엔 비적용).

### 1.2 본문 구조 — `ref_type=function` 템플릿 (함수 레퍼런스)

H2 섹션 본문은 **명명 블록**으로 작성. 빌더는 이 블록을 파싱해 구조화 JSON(§3)으로 변환. 누락 블록은 §5 코드로 보고.

| 블록 마커(H3 텍스트, 정확일치) | 필수 | 내용 계약 | 산출 필드 |
|------|:---:|------|------|
| `### 구문` | ✅ | 코드펜스 1개. `=FUNC(arg1, [arg2], ...)` Excel 표기. `[]`=선택인수 | `syntax.signature: string` |
| `### 인수` | ✅ | 표: `이름\|필수\|타입\|설명`. 행 수 ≥1 | `syntax.args: Arg[]` |
| `### 반환` | ✅ | 표 또는 문장. 타입(§1.2-T enum) + 한줄 설명 | `returns: {type, desc}` |
| `### 예제` | ✅ | 코드펜스 ≥1 + 각 예제 결과. N≥1 (결과 추출 = §1.2-E) | `examples: Example[]` |
| `### 실수포인트` | ✅ | 불릿 리스트 ≥1. 흔한 오류·감점 사유 | `pitfalls: string[]` |
| `### 유사함수대조` | ❌ | 표: `함수\|차이\|언제` 또는 `없음` 명시 | `contrast: Contrast[]` |

`Arg` 객체:

| 필드 | 타입 | 허용값 |
|------|------|------|
| `name` | string | Excel 인수명. 빈문자 금지 |
| `required` | boolean | 입력표의 `필수` 열(`Y`/`N` 또는 동등 표기) → 산출 boolean. 입력 표기는 작성 컨벤션, **산출 boolean만 계약** |
| `type` | string(enum) | §1.2-T |
| `desc` | string | 설명 |

§1.2-T **타입 enum**(Arg.type 및 returns.type 공통): `number` \| `text` \| `logical` \| `range` \| `array` \| `reference` \| `any`. 그 외 값 = `E_TYPE_ENUM`.

§1.2-E **예제 결과 추출 규칙**(결정적): `### 예제`의 각 코드펜스 직후 줄의 결과 표기에서 산출한다. 표기 형식 = `// => <result>` 한 줄 주석(코드펜스 마지막 줄 또는 펜스 바로 다음 줄). 주석 없으면 `result`는 빈 문자열 허용하되 `W_EXAMPLE_NO_RESULT`(경고). `note`는 결과 줄 뒤 추가 산문(옵션).

`Example` 객체: `{ formula: string, result: string, note?: string }`
`Contrast` 객체: `{ func: func_name, diff: string, when: string }`

### 1.3 본문 구조 — `ref_type=procedure` 템플릿 (절차 레퍼런스)

| 블록 마커(H3 정확일치) | 필수 | 내용 계약 | 산출 필드 |
|------|:---:|------|------|
| `### 단계` | ✅ | 번호 목록(순서有). 각 단계 = 한 동작. N≥1 | `steps: Step[]` |
| `### 메뉴경로` | ✅ | 표 또는 불릿. `리본탭 > 그룹 > 명령` 형식 | `menu_path: string[]` |
| `### 단축키` | ❌ | 표: `동작\|키`. 키 표기 §1.5. 없으면 `없음` 명시 | `shortcuts: Shortcut[]` |
| `### 주의` | ✅ | 불릿 ≥1. 함정·감점·전제조건 | `cautions: string[]` |

`Step` 객체: `{ no: integer, action: string, detail?: string }`
- `no`는 산출 시 빌더가 **출현 순서대로 1부터 재부여**한다(작성자 입력 번호는 표시 순서 신호일 뿐, 산출 `no`는 빌더 정규화). 따라서 작성자 입력 번호의 연속성은 강제하지 않는다.

`Shortcut` 객체: `{ action: string, keys: string }` (`keys`는 §1.5 정규화 권장 — 위반은 경고 `W_SHORTCUT_FORMAT`, 본문은 살림)

### 1.3-C 본문 구조 — `ref_type=concept` 템플릿 (개념 레퍼런스)

`ref_type=concept`는 구조 블록을 강제하지 않는 자유 본문이다. v1에서 **최소 구조 강제 없음**(§6.5 확정 — 개념 섹션은 서술형 레퍼런스이므로 블록 파싱 비적용). 본문 전체를 `ConceptBody.html`로 렌더한다(§3.1). function/procedure용 블록 검사(V3)는 concept에 적용하지 않는다.

### 1.4 영역(`area`/`subarea`) enum — 컴활1급 (자료조사 기반)

_인터페이스계약 §11: **2필드 {area, subarea}** 통일(점결합 `written.computer` 표기 폐기).

```
area     = written | practical            (필기 | 실기)
subarea  (area=written)   = computer | spreadsheet | database
         (area=practical) = excel | access
```

| {area, subarea} | 의미 |
|------|------|
| {written, computer} | 필기-컴퓨터 일반 |
| {written, spreadsheet} | 필기-스프레드시트 일반 |
| {written, database} | 필기-데이터베이스 일반 |
| {practical, excel} | 실기-스프레드시트 실무 |
| {practical, access} | 실기-데이터베이스 실무 |

§1.4-A **area↔subarea 정합 규칙**: `subarea`가 `area`의 허용집합에 속해야 한다(`area=written` ⇒ `subarea∈{computer,spreadsheet,database}`, `area=practical` ⇒ `subarea∈{excel,access}`). 위반 = `E_AREA_SUBAREA_MISMATCH`.

- `area`/`subarea` enum의 단일 SoT = **생성규칙.md config**(과목 영역 정의 소유). 본 문서는 컴활1급 현행값을 명시·참조하며, 변경 시 config 우선. 합격경로 진단(기획 §4)의 영역별 목표·판정 로직은 **엔진규격이 소유**. 개념서는 `area`/`subarea` 라벨만 제공(경계 준수: 합격확률예측 없음).

### 1.5 단축키 표기 정규화 규칙 (권장 — WARN 수준)

- 권장 키 토큰 예: `Ctrl` `Shift` `Alt` `Enter` `Tab` `Esc` `F1`..`F12` `Space` `Del` 및 단일 영숫자. (콘텐츠 영역이므로 토큰 집합은 닫힌 계약이 아닌 권장 — 추가 키/마우스동작은 위반이 아니며 경고 대상도 아님.)
- 조합 구분자 = `+` (공백 없음). 예 `Ctrl+Shift+Enter`, `Alt+F11`.
- 권장 정규식: `^([A-Za-z0-9]|Ctrl|Shift|Alt|Enter|Tab|Esc|Space|Del|F[1-9]|F1[0-2])(\+([A-Za-z0-9]|Ctrl|Shift|Alt|Enter|Tab|Esc|Space|Del|F[1-9]|F1[0-2]))*$`
- 위 권장 표기를 벗어나면 `W_SHORTCUT_FORMAT`(경고; 본문은 살리되 리포트). 빌드 차단 아님.

## 2. 개념서↔카드 양방향 링크 계약 (핵심)

기획 §3/§9: 섹션↔연결카드 바로풀기. 링크는 **양쪽에서 선언**되고 빌드시 **상호 무결성 검증**.

### 2.1 방향과 SoT

| 방향 | 선언 위치(SoT) | 필드 | 본 문서 역할 |
|------|------|------|------|
| 섹션 → 카드 | 개념서 front-matter | `related_cards: card_id[]` (_인터페이스계약 §12) | **소유·정의** |
| 카드 → 섹션 | 카드 데이터(카드규격) | `links.concept_ref: section_id` (단일, 0 또는 1) | **참조·검증규칙만** |

- 카드 1장은 `links.concept_ref` **0 또는 1개**(다대일). 섹션 1개는 `related_cards` **0..N개**. → 관계 = 섹션 1 : 카드 N.
- 카드측 역링크 필드명 = **`links.concept_ref`**(카드규격.md §5 SoT). 카디널리티 = 단일(0 또는 1). 본 계약의 R-LINK 검증식은 이 SoT를 따른다.
- 한 카드가 복수 섹션을 참조하는 실수요는 v1 스코프에서 **비채택**(§6.6 확정 — 단일 다대일 유지). INDEX+MATCH 같은 조합 카드는 대표 섹션 1개에 링크하고 보조 개념은 섹션 `prereq`/`related_cards` 그래프로 접속한다. 복수참조 승격은 v2 검토 사안.

### 2.2 무결성 규칙 (빌드 게이트, 위반시 에러코드)

1. **R-LINK-1 (역참조 존재)**: `section.related_cards`의 모든 `card_id`는 실제 카드로 존재 → 없으면 `E_DANGLING_CARD`.
2. **R-LINK-2 (정참조 존재)**: 카드의 `links.concept_ref`는 실제 `section_id`로 존재 → 없으면 `E_DANGLING_SECTION`.
3. **R-LINK-3 (양방향 일치)**: `card.links.concept_ref == S` 이면 `S.related_cards`에 그 `card_id` 포함, 역도 성립 → 불일치 `E_LINK_ASYMMETRY`.
   - **자동치유 옵션**: 빌더가 `--autolink` 플래그일 때 **양쪽 대상이 모두 실재(R-LINK-1·R-LINK-2 통과)하나 한쪽 선언만 누락**인 경우에 한해 다른쪽에 보강 후 `W_LINK_AUTOFILLED` 경고. **dangling(R-LINK-1/2 실패)은 autolink 치유 대상 아님** — 존재하지 않는 대상을 생성하지 않는다(에러 그대로). 기본 OFF(엄격).
4. **R-LINK-4 (다중참조 일관)**: 한 `card_id`는 둘 이상의 섹션 `related_cards`에 동시 등장 불가(카드는 `concept_ref` 단일이므로) → 위반 `E_LINK_MULTI_SECTION`.
5. **R-LINK-5 (deck 정합)**: 카드의 deck 네임스페이스와 섹션 `subject`가 동일 과목 → 불일치 `E_NS_MISMATCH`. _인터페이스계약 §2: deck/namespace는 card_id가 인코딩하지 않으며 카드의 `subject` 필드(+manifest)에서 온다("card_id에서 deck 추출" 규칙은 설계상 존재하지 않음). 따라서 비교 = 카드 `subject` vs 섹션 `subject` 직접 일치. (card_id 파싱 의존 없으므로 보류 불필요 — 즉시 활성.)

- **고립 섹션 허용**: `related_cards=[]` 이고 `prereq=[]` 인 섹션(연결 카드·선수 없는 독립 레퍼런스)은 **정상**(에러·경고 아님). 개념서는 카드 없는 순수 읽기 섹션을 가질 수 있다.

### 2.3 런타임 링크 산출 (정적 HTML + localStorage 경계)

빌더는 양방향 링크를 **정적 인덱스**로 산출(런타임 서버 조회 없음). 산출 계약 = 아래 두 매핑을 제공한다(자료구조의 키 이름·중첩 형태는 소비자 규격이 합의하는 구현 디테일):

- 섹션→카드: `Record<section_id, card_id[]>` (섹션 `related_cards` 기반)
- 카드→섹션: `Record<card_id, section_id>` (카드 `links.concept_ref` 기반)

- UI 계약: 개념서 섹션뷰 → 섹션→카드 매핑으로 "이 섹션 카드 바로풀기" 노출. 카드뷰 → 카드→섹션 매핑으로 "개념서에서 보기" 역링크. (UI 표현·동선·노출 키 구조는 UI규격 소유.)
- localStorage 비저장: 링크 인덱스는 빌드 정적물(코드/JSON)이며 학습이력이 아니므로 localStorage에 넣지 않음(쿼터 절약). localStorage엔 오답/박스/진도만(엔진규격 소유).

## 3. 빌드 산출 스키마 (generate.py 출력 — 무의존 stdlib JSON)

개념서 빌드 산출 = 과목 deck에 동봉되는 `concepts` 레지스트리(배열). 전역 script 주입(file:// fetch 회피, 기획 §5). **등록 네임스페이스 키·전역 변수명은 생성규칙.md/엔진규격.md 소유** — 본 문서는 `concepts` 배열의 원소 스키마만 계약한다.

### 3.1 ConceptSection (산출 객체)

| 필드 | 타입 | 출처 |
|------|------|------|
| `section_id` | string | front-matter |
| `subject` `area` `subarea` `unit` `title` | string | front-matter |
| `ref_type` | enum | front-matter |
| `weight` | float [1,10] | front-matter |
| `anchor` | string | §0 결정적 파생 |
| `related_cards` | card_id[] | front-matter(검증후) |
| `prereq` | section_id[] | front-matter |
| `body` | FunctionBody \| ProcedureBody \| ConceptBody | §1.2/1.3/1.3-C 파싱 |
| `schema_version` | int | front-matter |

- `FunctionBody` = `{ syntax:{signature:string, args:Arg[]}, returns:{type:string, desc:string}, examples:Example[], pitfalls:string[], contrast:Contrast[] }`
- `ProcedureBody` = `{ steps:Step[], menu_path:string[], shortcuts:Shortcut[], cautions:string[] }`
- `ConceptBody` = `{ html:string }` (`ref_type=concept`는 자유 본문; 렌더 HTML이 본문 SoT — 최상위 별도 html 필드 없음)

> 주: 섹션 렌더 HTML은 `body`(특히 ConceptBody) 내부에 둔다. ConceptSection 최상위에 별도 `html` 필드를 두지 않는다(SoT 단일화).

### 3.2 html 렌더 계약 (결정적·무의존)

`body`의 렌더 HTML은 generate.py가 stdlib만으로 생성하며 **결정적**(동일 입력→동일 출력, V10)이어야 한다.

- 허용 태그 화이트리스트: `h2 h3 h4 p ul ol li table thead tbody tr th td code pre strong em a span br`. 그 외 태그·원시 HTML은 이스케이프(텍스트로). 위반 입력 = 살리되 `W_HTML_STRIPPED`.
- 허용 속성: `a[href]`(내부 앵커 `#<anchor>` 또는 동일 과목 `section_id` 링크만), `td/th[align]`, `code/pre[class="lang-*"]`. 그 외 속성 제거.
- 변환 범위: 마크다운 부분집합(문단·불릿/번호 목록·표·인라인 코드·코드펜스·강조·링크)만. 스크립트/이벤트 핸들러/외부 URL(`http(s)://`) `href` 금지(file:// 정적 경계).
- 인코딩: UTF-8, no BOM. 직렬화 = stdlib `json` only(무의존).

### 3.3 anchor 파생·충돌 계약

- `anchor`는 `section_id`에서 결정적으로 파생한다(파생 알고리즘 = 빌더 구현 자유: 예 `.`→`-` 치환, 슬러그화 등).
- **충돌 검사(계약)**: 서로 다른 `section_id`가 동일 `anchor`로 파생되면 빌드 에러 `E_ANCHOR_COLLISION`. 빌더는 충돌 시 결정적 디스앰비규에이션(접미 일련번호 등)을 적용해도 되나, 결과는 결정적이어야 한다(V10).

### 3.4 산출 배열 정렬

- `concepts` 배열은 `section_id` 오름차순(소문자 비교) 안정정렬로 산출한다(결정성·V10 근거).

## 4. 검증 게이트 (generate.py가 빌드시 실행 — 무의존 테스트셋 동반)

전 검사를 실행하고 **에러 집합**을 산출한다(실행 순서·병렬·조기중단은 빌더 구현 자유 — 계약은 결과 집합). 하나라도 ERROR면 빌드 실패(exit≠0). WARN은 리포트 후 진행.

| # | 검사 | 위반 레벨/코드 |
|---|------|------|
| V1 | front-matter 필수필드 존재·타입·길이(`unit`≤40,`title`≤60) | ERROR `E_SCHEMA` |
| V2 | `section_id` 정규식·과목내 유일 | ERROR `E_DUP_ID` / `E_ID_FORMAT` |
| V3 | `ref_type=function`/`procedure` 필수 블록 존재(concept 제외) | ERROR `E_MISSING_BLOCK` |
| V4 | Arg/returns `type` enum(§1.2-T) | ERROR `E_TYPE_ENUM` |
| V5 | 단축키 표기 정규화 | WARN `W_SHORTCUT_FORMAT` |
| V6 | `func_name`(function 섹션) 정규식 | WARN `W_FUNC_NAME` |
| V7 | 링크 무결성 R-LINK-1..5 | ERROR 각 코드(§2.2) / autolink시 WARN `W_LINK_AUTOFILLED` |
| V8 | `prereq` 존재(`E_DANGLING_PREREQ`) + 사이클 없음(`E_CYCLE`) | ERROR |
| V9 | `weight` 범위 float [1,10] | ERROR `E_WEIGHT_RANGE` |
| V10 | 빌드 결정성: 동일 입력→동일 산출(anchor 충돌·정렬 안정) | ERROR `E_NONDETERMINISTIC` / `E_ANCHOR_COLLISION` |
| V11 | `related_cards` 중복·`tags` 형식/개수·`schema_version` 값 | ERROR `E_DUP_LINK` / WARN `W_TAG_FORMAT` / ERROR `E_SCHEMA_VERSION` |
| V12 | `area`↔`subarea` 정합(§1.4-A) | ERROR `E_AREA_SUBAREA_MISMATCH` |

- 검증기는 **테스트셋 동반 필수**(기획 §5): 정상 1섹션 + 의도적 위반 케이스(각 에러코드 1개 이상) 픽스처로 회귀. 테스트셋 위치/형식은 검증 게이트 산출과 합의.
- 리포트: **콘솔 직접 print 금지**(cp949 한글 깨짐 회피), UTF-8 파일로 산출. 파일명·경로는 구현/검증게이트 영역(본 계약은 "비콘솔·UTF-8" 채널 의도만 고정).

## 6. 결정 사항 (확정) 및 잔여 [OPEN]

확정(이 계약 본문에 반영됨):
- **D1 (섹션 단위)**: 파일 1개 = 섹션 1개(§1). 파일 내 H2 다중분할 비채택.
- **D2 (weight 산정)**: 작성자 수동 명시(필수, float [1,10] — _인터페이스계약 §6). 자동 ★매핑은 가이드만, 개념서는 필수 강제(§1.1).
- **D3 (역링크 필드)**: 카드측 = `links.concept_ref`(단일, 0/1). 카드규격.md SoT(§2.1).
- **D4 (subject_ns 컴활1급)**: `comp` 확정(§0,§1.1).
- **D5 (concept 본문)**: ref_type=concept 최소 구조 강제 없음, 자유 본문(§1.3-C).
- **D6 (복수 섹션 참조)**: v1 비채택, 단일 다대일 유지(§2.1). v2 검토.
- **D7 (area SoT)**: 생성규칙.md config 소유(2필드 {area, subarea}, _인터페이스계약 §11), 개념서는 라벨 참조·진단 목표는 엔진규격(§1.4).
- **D8 (deck 추출 — 확정/_인터페이스계약 §2)**: card_id는 namespace를 인코딩하지 않으며 "card_id에서 deck 추출" 규칙은 설계상 **존재하지 않음**. deck = 카드 `subject` 필드(+manifest). R-LINK-5는 카드 `subject` vs 섹션 `subject` 직접 비교로 **즉시 활성**(이전 보류·`W_NS_UNCHECKED` 폐기, Q-CARD-DECK 종결).
