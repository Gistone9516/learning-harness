# 생성규칙 규격 (계약)

> buildflow ② 명세. 본 문서는 **기계검증 가능한 계약**이다(자유서술 아님).
> 범위: 콘텐츠 md → deck + manifest 변환 규칙 · 전역 script 출력(file:// fetch 회피) · 과목 config 스키마 · config-driven 엔진 계약(엔진 무오염) · export/백업 포맷.
> 비범위(타 규격 소관): 카드 필드 의미·5유형 정의 → `카드규격.md` / Leitner·채점코어·localStorage 런타임 스키마 → `엔진규격.md` / UI 토큰 → `UI규격.md` / 개념서 섹션 id → `개념서규격.md`.
> 불변경계(기획_v4 수렴): 정적HTML+localStorage(서버0) · 무의존 stdlib · 자동채점 없음(=`self` 모드는 사용자 O/X를 그대로 반영, 앱이 정오 판정 안 함) · 합격확률 예측 없음 · 엔진 무오염.
>
> **용어 정의 — "자동채점 없음"의 정확한 의미**: 불변경계의 "자동채점 없음"은 **실기 손풀이(빈 화면 클릭경로 재현)를 시뮬레이션/측정하지 않는다**는 뜻이다(기획_v4 §2). exact/keyword/cloze의 텍스트 기계비교(인출 응답 정오 판정)는 자동채점에 해당하지 않으며 허용된다. `self` 모드만 기계비교를 하지 않고 사용자 O/X를 채택한다.

## 0. 용어·식별자 규약

- `subject_id`: 과목 네임스페이스. 정규식 `^[a-z][a-z0-9_]{1,31}$`. 예: `comp_act_1`.
- `deck_id`: 덱(콘텐츠 파일 1개 = 덱 1개) 식별자. 정규식 `^[a-z][a-z0-9_]{1,63}$`. `subject_id` 내 유일.
- `card_id`: **작성자 명시 안정 id**(content-hash 금지 — 오탈자 수정에도 오답이력 보존). 정규식 `^[a-z][a-z0-9-]{2,63}$`(소문자 시작, 소문자·숫자·하이픈, 3~64자. 대문자·언더스코어·콜론 금지 — _인터페이스계약 §1). 하이픈 세그먼트 강제 안 함. **유일성 범위 = deck namespace(subject_id) 내**. 전역 유일성은 `subject_id + ":" + card_id` 조합으로 성립(문자열 단독 전역유일 강제 안 함).
- `section_id`: 개념서 섹션 id(개념서규격 소관). 본 규격에서는 카드↔섹션 링크 참조키로만 취급. 정규식은 개념서규격 소관.
- 모든 id 비교는 **대소문자 구분(case-sensitive)**.
- 인코딩: 모든 입출력 파일 UTF-8(BOM 없음), 개행 LF.

### 0.1 과목 폴더 ↔ subject_id 해소 규칙
- 입력/출력 경로의 `<subject_label>`는 **사람용 폴더 표시명**(예: `컴활1급`)으로, 식별자가 아니다.
- `subject_id`(기계용 식별자)는 **오직 config 파일과 front-matter의 `subject_id` 필드**에서만 가져온다. 폴더명에서 추론하지 않는다.
- 빌드 진입 시 `python generate.py build --subject <subject_id>`로 `subject_id`를 명시하고, 그 값으로 `config/<subject_id>.config.js`를 찾는다. config의 `subject_label`이 곧 출력 경로의 `<subject_label>`이다(폴더 표시명 = config가 진실).
- `subject_label`은 임의 표시 문자열이며 경로로 쓸 때 OS 안전화는 §0.2를 따른다.

### 0.2 파일시스템 안전 규약 (case-insensitive FS 충돌 방지)
- `card_id`(정규식상 소문자 전용)·`deck_id`·`subject_id`는 비교상 case-sensitive(§0)이나 출력 파일명(`decks/<deck_id>.js`)은 Windows 등 대소문자 무시 파일시스템에서 충돌할 수 있다.
- 규칙: 한 과목 내 `deck_id` 집합을 **소문자로 접은(lower-cased) 값이 충돌**하면 `E_DECK_ID_CASE_COLLISION`. (deck_id 정규식은 이미 소문자만 허용하므로 §0 준수 시 충돌 불가하나, 미래 정규식 변경 대비 검증을 계약으로 둔다.)
- 출력 파일명은 `deck_id`를 그대로 쓴다(추가 안전화 변환 없음 — 정규식이 안전 문자만 허용).

## 1. 입력 — 콘텐츠 md (덱 소스)

### 1.1 파일 위치·매핑
- 입력 루트: `과목/<subject_label>/콘텐츠/*.md`.
- 파일 1개 = 덱 1개. `deck_id` = 파일 front-matter의 `deck_id` 필드(파일명 아님 — 파일명 변경에 id 불변).

### 1.2 덱 파일 구조(계약)
파일 = `① front-matter 블록` + `② 카드 블록 0..N개`.

**front-matter 파싱 계약 (stdlib-only)**: front-matter는 YAML 전체가 아니라 **아래로 제한된 플랫 키-값 서브셋**이며, 자작 라인 파서(stdlib `re`)로 파싱한다(3rd-party YAML 금지, §7).
- 파일 선두 `---` 라인으로 시작, 다음 `---` 라인으로 종료.
- 각 라인은 정확히 `key: value`(콜론+스페이스 1개 구분). 중첩·블록·앵커·다중행 값 미지원.
- `value`는 종료개행 전까지의 문자열. 양끝 공백 trim. 따옴표(`"..."`)로 감싸면 따옴표 제거 후 내부를 리터럴로 취급(콜론 포함 가능). 따옴표 없으면 첫 콜론 이후 전체가 value(추가 콜론 허용).
- 빈 라인·`#`로 시작하는 주석 라인 무시.
- 형식 위반(`key: value` 패턴 불일치, 종료 `---` 누락) → `E_FRONTMATTER_INVALID`.

front-matter 필드:

| 필드 | 타입 | 필수 | 허용값/제약 | 기본값 |
|---|---|---|---|---|
| `deck_id` | string | Y | §0 정규식 | — |
| `subject_id` | string | Y | §0 정규식, config의 subject_id와 일치 | — |
| `title` | string | Y | 1..120자 | — |
| `unit` | string | Y | 단원 id. 작성자 공급 → generate.py가 deck Card.unit으로 통과(_인터페이스계약 §14) | — |
| `area` | enum | Y | `written` \| `practical` (2값 — _인터페이스계약 §11) | — |
| `subarea` | enum | Y | area=`written`: `computer`\|`spreadsheet`\|`database` / area=`practical`: `excel`\|`access` (_인터페이스계약 §11) | — |
| `schema_version` | int | Y | `1` (현재) | — |

- `schema_version` 값이 정수가 아니거나 1 미만 → `E_FRONTMATTER_INVALID`.
- `schema_version > 1`(코드가 모르는 미래 버전) → `E_SCHEMA_UNSUPPORTED`(빌드 중단, 다운그레이드 금지).

### 1.3 카드 블록 문법(계약)
- 카드 시작 마커: 줄 시작 `## @<card_id>` (정규식 `^##\s+@(\S+)\s*$`). 캡처된 `<card_id>`는 추가로 §0 `card_id` 정규식을 충족해야 한다. 마커엔 매치되나 §0 정규식 위반 시 → `E_CARD_ID_INVALID`.
- 카드 메타: 시작 마커 다음, 첫 본문/분리마커 전까지의 `- key: value` 라인(들). value 파싱은 §1.2 front-matter와 동일 규칙(콜론+스페이스, 따옴표 옵션, trim). 허용 key:

| key | 타입 | 필수 | 허용값 | 기본값 |
|---|---|---|---|---|
| `type` | enum | Y | `func` \| `proc` \| `recall_seq` \| `cloze` \| `judge` (이 5개 토큰만 — _인터페이스계약 §3) | — |
| `unit` | string | N | 단원 id. 카드 단위 재정의 시 명시. 미지정 시 덱 front-matter `unit` 상속. 작성자 공급 → generate.py가 deck Card.unit으로 통과(_인터페이스계약 §14) | 덱 front-matter `unit` |
| `grade_mode` | enum | N | `exact` \| `keyword` \| `cloze` \| `self` | §4.3 type별 기본 |
| `area` | enum | N | `written` \| `practical` (_인터페이스계약 §11) | 덱 front-matter `area` |
| `subarea` | enum | N | area 값에 따른 subarea enum(§1.2) | 덱 front-matter `subarea` |
| `weight` | float | N | `[1, 10]` (빈출·배점 가중 — _인터페이스계약 §6). 범위 밖 → 클램프, 비숫자 → 5 폴백 | `5` |
| `concept_ref` | string | N | 개념서 섹션 id(카드→개념 링크, _인터페이스계약 §12). 산출 Card의 `links.concept_ref` 필드로 직렬화 | null |
| `tags` | string[] | N | `,`(콤마) 구분 CSV, 각 토큰 trim 후 `^[A-Za-z0-9_-]+$`. **authoring 전용 플랫 입력**. generate.py가 빌드 시 산출 Card의 `tags` 오브젝트({weight,area,subarea})로 조립(_인터페이스계약 §14). | `[]` |
| `answers` | string[] | exact·recall_seq일때 Y | `\|` 구분 복수정답(exact: OR 집합, recall_seq: 순서 보존 배열), 각 토큰 trim 후 1자 이상. **authoring 소스 필드**. generate.py가 빌드 시 `answer_spec`으로 조립(→ §2.4). | — |
| `keywords` | string[] | keyword일때 Y | `\|` 구분, 각 토큰 trim 후 1자 이상. **authoring 소스 필드**. generate.py가 빌드 시 `answer_spec`으로 조립(→ §2.4). | — |

- 리스트 value(`tags`/`answers`/`keywords`) 토큰 분리 후 빈 토큰(공백뿐) 발생 → 해당 코드 에러(§6): answers 빈토큰 → `E_MISSING_ANSWERS`, keywords 빈토큰 → `E_MISSING_KEYWORDS`, tags 빈/정규식위반 토큰 → `E_TAG_INVALID`.
- 알 수 없는 메타 key → `E_CARD_META_UNKNOWN`.
- 카드 본문: 메타 라인 종료 후 다음 카드 마커(또는 EOF) 전까지.
  - 앞면/뒷면 분리 마커: `---FRONT---` / `---BACK---` (각 줄 단독). FRONT=최소단서, BACK=상세.
  - **분리 마커 배치 규칙**: 본문에 `---FRONT---`가 1회, 그 뒤에 `---BACK---`가 1회, 이 순서로 정확히 등장해야 한다. front = `---FRONT---`~`---BACK---` 사이, back = `---BACK---`~카드끝.
    - 마커가 0개(분리 없음): front/back 모두 비게 되므로 `E_FRONT_BACK_MISSING`.
    - 마커 중복(`---FRONT---` 또는 `---BACK---`가 2회 이상)·순서역전(`---BACK---`가 `---FRONT---`보다 먼저)·한쪽만 존재 → `E_FRONT_BACK_MALFORMED`.
    - front 또는 back 영역이 trim 후 빈 문자열 → `E_FRONT_BACK_EMPTY`.
  - cloze 빈칸 토큰: `{{정답}}` 또는 `{{정답|별칭|별칭}}`. (cloze 파싱 의미규칙 상세 = 엔진규격 소관; 본 규격은 토큰 문법·산출 형태만 계약, §2.4·§7.)
    - 리터럴 중괄호: 본문에서 `{` `}` 자체를 표기하려면 `\{` `\}`로 이스케이프한다. 이스케이프된 중괄호는 cloze 토큰 스캔에서 제외하고, 산출 front 문자열에는 `{`/`}`로 환원한다.
  - **cloze 토큰 ↔ type 정합**: `type:cloze`이면 본문(front)에 cloze 토큰 1개 이상 필수(0개 → `E_CLOZE_NO_BLANK`). `type:cloze`가 아닌 카드의 본문에 cloze 토큰이 나타나면 → `E_CLOZE_UNEXPECTED`.
- type↔grade_mode 적합성: §4.3 매트릭스 위반 시 §6 에러 `E_GRADE_TYPE_MISMATCH`.

### 1.4 검증 규칙(빌드 시 강제)
- `card_id` deck namespace(subject_id) 내 중복 → `E_CARD_ID_DUP`(_인터페이스계약 §1: 전역 문자열 유일이 아니라 deck 범위 유일).
- `deck_id` 과목내 중복 → `E_DECK_ID_DUP`.
- `concept_ref` 참조가 개념서 인덱스에 없음 → **경고** `W_CONCEPT_REF_DANGLING`(빌드 계속, _인터페이스계약 §12: 개념서 미동반 시 보류). 개념서 인덱스가 빌드 시점에 제공되지 않으면(파일 부재) → 참조 검증 자체를 **스킵**하고 경고 `W_SECTION_INDEX_ABSENT` 1회 출력(빌드 계속). 인덱스 경로/포맷은 개념서규격 의존(§10 O1).
- front-matter/카드 `area` ∉ {`written`,`practical`} 또는 `subarea`가 해당 area의 허용 enum 밖 → `E_AREA_UNKNOWN`(_인터페이스계약 §11).
- front-matter `subject_id` ≠ `--subject` 인자값 → `E_SUBJECT_ID_MISMATCH`.

## 2. 출력 — deck + manifest (file:// fetch 회피)

### 2.1 핵심 계약: 전역 script 출력
- **JSON 파일을 fetch하지 않는다**(`file://` 프로토콜에서 `fetch`/XHR이 CORS·로컬파일 정책으로 차단되므로). 대신 **각 산출물을 전역 변수에 할당하는 `.js`** 로 출력하고 HTML이 `<script src>`로 동기 로드한다.
- 무로드 과목전환: 모든 deck.js·manifest.js는 정적 script로 사전 로드 가능해야 하며, 전역 레지스트리에 자기등록한다.
- **관찰가능 계약**: 로드 후 전역 객체(`window.__DECKS__`/`window.__MANIFEST__`/`window.__SUBJECT_CONFIG__`)가 §2.3~§2.5·§3.1 구조를 갖는다. 그 등록을 수행하는 **코드 형태(IIFE·할당문·기타)는 구현 자유**다.

### 2.2 출력 위치
- 루트: `과목/<subject_label>/생성물/`.
- `manifest.js` (과목당 1개)
- `decks/<deck_id>.js` (덱당 1개)
- **스테일 산출물**: 현 빌드의 manifest.decks에 없는데 `decks/`에 잔존하는 `*.js`는 **경고** `W_STALE_DECK_FILE`(파일 나열). 자동 삭제는 안 함(`--clean` 옵션 시에만 삭제, §7).

### 2.3 deck.js 포맷(계약)
deck.js는 로드 시 전역 레지스트리에 자기 등록한다(구현 형태 자유, §2.1). 관찰가능 결과는 아래 객체가 `window.__DECKS__["<deck_id>"]`에 존재함이다.

```js
// 예시(IIFE/할당문 등 구현 자유) — 관찰가능 계약은 등록 결과 객체 구조뿐
(window.__DECKS__ = window.__DECKS__ || {})["<deck_id>"] = {
  schema: 1,
  deck_id: "<deck_id>",
  subject_id: "<subject_id>",
  title: "<title>",
  cards: [ /* Card 객체 배열, §2.4 — 아래 구조 예시:
    {
      card_id: "func-vlookup-basic",
      subject: "<subject_id>",
      unit: "<unit_id>",
      type: "func",
      grade_mode: "exact",
      front: { ... },
      back: { detail: "...", note: null },
      answer_spec: { accepted: ["VLOOKUP"], normalize: ["trim","lower"] },
      tags: { weight: 7, area: "practical", subarea: "excel" },
      links: { concept_ref: "sec-vlookup-intro" },
      enabled: true
    }
  */ ]
};
```

- 전역 네임스페이스 키: `window.__DECKS__` (object, key=deck_id).
- `schema`: int, 현재 `1`.
- **JS 문자열 리터럴 이스케이프(필수 안전 계약)**: 모든 문자열 필드(title/front/back/answer_spec 내 문자열 등) 직렬화 시 generate.py는 `json.dumps(value, ensure_ascii=False)` 결과를 JS 문자열로 사용한다(JSON 문자열은 유효한 JS 문자열). 추가로 `</script>`로 조기종료되지 않도록 `<`를 `<`로, 행분리자 U+2028/U+2029를 각각 ` `/` `로 치환한다. 이로써 따옴표·개행·백슬래시·`</script>`·유니코드 분리자가 모두 안전.

### 2.4 Card 객체 스키마(deck.js 내, 계약)

| 필드 | 타입 | 필수 | 비고 |
|---|---|---|---|
| `schema_version` | int | Y | 마이그레이션용. 현재 `1`. generate.py 빌드 산출(작성 대상 아님 — _인터페이스계약 §14) |
| `card_id` | string | Y | §0 정규식(`id` 금지 — _인터페이스계약 §14) |
| `subject` | string | Y | deck namespace 키(subject_id — _인터페이스계약 §14) |
| `unit` | string | Y | 단원 id. authoring 공급(§1.2 front-matter 또는 §1.3 카드 메타) → generate.py 통과(_인터페이스계약 §14) |
| `type` | enum | Y | §1.3 |
| `grade_mode` | enum | Y | §4.3 (상속·기본 해소 완료. **유일 이름 = `grade_mode`**, `grade`·`score_mode` 폐기 — _인터페이스계약 §14) |
| `front` | object | Y | type별 키(prompt/text/scenario/options). 렌더용(마크다운 인라인 허용 범위 = UI규격 소관) |
| `back` | object | Y | {detail?, note?} |
| `answer_spec` | object\|null | Y | 채점 데이터 객체. generate.py가 authoring 소스(answers/keywords/cloze 토큰)로부터 빌드 시 조립. grade_mode=self → **null**. non-nullable 선언 금지(_인터페이스계약 §14). §2.4.1 참조. |
| `tags` | object | Y | `{weight: float[1,10] 기본5, area: enum, subarea: enum}`. area·subarea 상속 해소 완료. weight 클램프·폴백 해소 완료. **최상위 직렬화 금지** — 반드시 tags 오브젝트 안(_인터페이스계약 §14) |
| `links` | object | Y | `{concept_ref?: string}`. 카드→개념 링크. 미동반 시 concept_ref 키 생략. `related_cards`는 개념서 필드(카드 links에 두지 말 것 — _인터페이스계약 §14) |
| `enabled` | bool | Y | 기본 true(_인터페이스계약 §14) |

- 빌드는 상속·기본값을 **모두 해소한 후** 직렬화한다(런타임이 다시 계산 안 함 — 엔진 무오염).
- **`answer_spec`은 `object|null`**: grade_mode=self → `null`. 그 외 grade_mode → `answer_spec` object 필수. 엔진은 `answer_spec`만 소비(평면 authoring 키는 deck Card에 노출 안 됨 — _인터페이스계약 §13).
- **중첩 구조 강제**: `weight`/`area`/`subarea`는 `tags` 오브젝트 안, `concept_ref`는 `links` 오브젝트 안. 최상위 직렬화 금지(_인터페이스계약 §14).
- **authoring 소스 vs 산출 deck**: 콘텐츠 md의 `answers`/`keywords`/cloze 토큰은 authoring 입력이며, deck Card에는 `answer_spec`만 직렬화한다. 두 표현을 혼용하지 않는다.

#### 2.4.1 answer_spec 객체 스키마 (generate.py 조립 — _인터페이스계약 §13)

generate.py는 빌드 시 카드의 authoring 소스 필드(answers/keywords/cloze 토큰)를 읽어 아래 type별 매핑으로 `answer_spec`을 조립한다. **엔진은 `answer_spec`만 소비**한다.

| type | grade_mode | answer_spec 필드 | 조립 규칙 |
|---|---|---|---|
| `func` | `exact` | `accepted: string[]` | `answers` 토큰(OR 집합). `grading.accept`가 있으면 합집합 |
| `judge` | `exact` | `accepted: string[]` | 동상 |
| `proc` | `keyword` | `requiredKeywords: string[][]` | `keywords` 토큰 배열. 내부 배열 = 한 필수그룹의 동의어 후보(any-of), 모든 그룹 필수(v1) — _인터페이스계약 §14 |
| `recall_seq` | `exact` | `sequence: string[]` | `answers` 토큰, **입력 순서 보존**(순서 의미) |
| `cloze` | `cloze` | `blanks: string[][]` | 본문 cloze 토큰을 0-base 순서로 변환. `blanks[i]` = i번 빈칸(위치=0-base 인덱스)의 정답 후보 배열. 별도 idx 필드 없음(위치가 인덱스 — _인터페이스계약 §14) |
| 임의 | `self` | **null** | grade_mode=self → `answer_spec: null`(_인터페이스계약 §14: self면 null, 키 부재 아님) |

**`blanks` 구조** (cloze 전용, _인터페이스계약 §13·§14):
- `answer_spec.blanks: string[][]` — **배열 인덱스 = 빈칸 위치(0-base)**. `blanks[0]`=첫 빈칸, `blanks[1]`=두 번째 빈칸 …
- `blanks[i]`는 해당 빈칸의 정답 후보 배열(OR). `{{정답|별칭1|별칭2}}` → `["정답","별칭1","별칭2"]`. 각 토큰 trim, 빈 토큰 불가(빈 토큰 → `E_CLOZE_NO_BLANK`).
- **`idx` 객체 필드 폐기**: 구 `{idx, accept}` 객체 배열 형식은 사용하지 않는다. 위치는 배열 인덱스로 자명.
- 채점 단위는 **빈칸**(빈칸별 정/오). 카드 단위 합산은 엔진규격 소관.

**`answer_spec.normalize`** (모든 type 공통):
- `normalize: string[]` — 이 카드에 적용할 정규화 규칙 id 순서배열(§3.4 카탈로그 기준). 엔진이 그대로 수용. 미지정 시 config type별 기본 프로파일 적용(_인터페이스계약 §14).

### 2.5 manifest.js 포맷(계약)

```js
// 예시 — 관찰가능 계약은 등록 결과 객체 구조뿐(구현 형태 자유)
(window.__MANIFEST__ = window.__MANIFEST__ || {})["<subject_id>"] = {
  schema: 1,
  subject_id: "<subject_id>",
  subject_label: "<표시명>",
  generated_at: "<ISO8601 UTC>",
  generator_version: "<semver>",
  areas: [ { area:"<written|practical>", subarea:"<subarea enum>", label:"<표시명>", target:<number|null> } ],
  decks: [
    { deck_id:"<id>", title:"<t>", area:"<written|practical>", subarea:"<subarea>", card_count:<int>, src:"decks/<deck_id>.js" }
  ],
  config_ref: "config/<subject_id>.config.js",
  counts: { cards_total:<int>, by_type:{...}, by_area:{...} }
};
```

- `generated_at`: ISO8601, UTC, 초 단위. `--reproducible` 빌드 시 고정값 `"1970-01-01T00:00:00Z"`로 출력(재현빌드 — §7·§10 O3 확정).
- `areas`: config.areas를 그대로 전파({area,subarea}·label·target 동일). target 파생 없이 config 값 복사(§3.3).
- `src`: manifest 기준 상대경로. HTML 로드 순서 = config.js → manifest.js → 각 deck.js → 엔진(§2.6).
- **삼자 subject_id 일치**: deck.js.subject_id == manifest.subject_id == config.subject_id == front-matter.subject_id. 불일치 → `E_SUBJECT_ID_MISMATCH`.

### 2.6 HTML 로드 계약(생성 책임 경계)
- generate.py는 script 태그 목록을 산출(또는 index.html 갱신). 순서 불변식:
  1. `config/<subject_id>.config.js`
  2. `manifest.js`
  3. `decks/*.js` (manifest.decks 순서)
  4. 엔진 코어 js (엔진규격 소관)
- 위반(엔진이 먼저 등) 시 런타임 미정의 → 빌드가 순서 보장 책임짐.
- generate.py는 **위 순서의 script 태그 목록(텍스트)을 산출**하는 것까지를 계약한다. 이 목록을 index.html에 주입하는 방식(전담 생성/템플릿 치환)은 §10 O1 잔여(개념서·엔진규격과 합의 대기 — escalate 아님, 산출물 형태만 다름).

### 3.4 normalize(채점 정규화 — exact/keyword 입력 전처리)

- 모델 = **순서 있는 규칙 id 배열** `normalize: string[]` + 공유 카탈로그(_인터페이스계약 §7 권위). 배열 **순서대로** 적용(= 계약). 기존 boolean 플래그 객체(NormalizeOptions) 폐기.
- 공유 카탈로그(id → 동작, 정의는 카드규격 §3.5 카탈로그가 SoT):
  `nfkc` · `trim` · `collapse_space` · `strip_all_space` · `lower` · `fullwidth_to_halfwidth` · `unify_cell_dollar` · `unify_arg_sep` · `strip_trailing_paren`
- config `normalize`는 **id 배열**(string[]). 카탈로그 외 id → `E_CONFIG_NORMALIZE_UNKNOWN`. 빈 배열 허용(정규화 없음).
- 카드 `grading.normalize`(id 리스트)가 있으면 엔진이 그대로 수용하고, 미지정 시 config의 type별 기본 프로파일(id 리스트)을 적용한다.
- 정규화는 **순수 함수 명세**: 같은 입력→같은 출력(결정성). **엔진 코어는 이 id 배열만 받아 동작**(과목 지식 하드코딩 금지 = 무오염).
- **동치 골든셋이 계약**: 본 규격은 정규화의 **입출력 동치(§8 골든셋)** 를 계약한다. 단 모델이 순서 있는 id 배열이므로 배열 순서대로 적용한 결과가 골든셋이다.
- **synonyms 치환 위치(cross-contract 확정)**: 동의어→대표어 치환은 **`lower` 적용 직후·정답 비교 직전**에 수행한다. 치환 단위는 토큰(공백 분리) 정확일치. 이 위치는 카드규격·엔진규격과 동일하게 고정한다(_인터페이스계약 §8).

### 3.5 grade_modes(채점모드 매핑 오버라이드)
- 형식: `{ "<type>": "<grade_mode>" }`. type별 기본 grade_mode를 과목 단위로 바꿀 때만 사용.
- 우선순위: 카드별 `grade_mode` 명시 > config `grade_modes` > §4.3 기본.
- config가 오버라이드한 grade_mode도 §4.3 type↔grade_mode 허용 매트릭스를 만족해야 한다. 위반 → `E_CONFIG_GRADE_TYPE_MISMATCH`.

### 3.6 LeitnerConfig(파라미터 주입 — 코어 무오염)

| 필드 | 타입 | 제약 | 기본값 |
|---|---|---|---|
| `boxes` | int | 3..7 | 3 |
| `intervals_days` | int[] | **길이 = boxes**, 양의 정수 오름차순(박스별 1대1 간격) | `[1,3,7]` |
| `dday_compress` | bool | t/f | true |
| `promote_on` | enum | `cold_first_correct` 고정 | `cold_first_correct` |

- `boxes` 기본 `3`, `intervals_days` 기본 `[1,3,7]`(길이 3) — 기본값끼리 정합(기획_v4 §3: 1/3/7일 3박스).
- `intervals_days` 길이 ≠ `boxes` → `E_CONFIG_INTERVALS_LEN`. (박스별 1대1 대응으로 확정 — 엔진규격 Leitner와 동기화.)
- `promote_on`은 기획 §3 결정(세션간 cold 첫시도 정답만 승급) 고정값. 다른 값 금지 → `E_CONFIG_PROMOTE_INVALID`.

### 3.7 config 검증
- `subject_id` ≠ 파일명 stem(`<subject_id>.config.js`의 `<subject_id>`) → `E_CONFIG_ID_MISMATCH`.
- `synonyms`에 빈 배열/빈 문자열 대표어/빈 동의어 → `E_CONFIG_SYNONYM_EMPTY`. (작성 형식 `{대표어:[동의어,...]}`은 generate.py가 역인덱스 `Record<동의어,대표어>`로 컴파일 — §4.2·_인터페이스계약 §8.)
- `normalize` 배열에 §3.4 카탈로그 외 id → `E_CONFIG_NORMALIZE_UNKNOWN`.
- `grade_modes` 값이 grade_mode enum 외 → `E_CONFIG_GRADE_UNKNOWN`.
- `grade_modes`/`areas`/`leitner` 등 §3.2 제약 위반 → 해당 코드(§6).

## 4. config-driven 엔진 계약(엔진 무오염)

### 4.1 무오염 불변식(계약)
- `엔진/` 코어 코드(js·py)는 **과목 id·과목 어휘·동의어·정규화 분기·area 키를 하드코딩하지 않는다**. 모든 과목 차이는 §3 config 객체 주입으로만 표현.
- 검증: 엔진 코어 소스에 등록된 임의 `subject_id` 문자열 리터럴이 등장하면 위반(grep 게이트 §9).

### 4.2 엔진 진입 인터페이스 시그니처(경계 계약)
> 아래 시그니처는 **본 규격이 엔진에 넘기는 데이터 계약(인자 형태)** 만 고정한다. 채점 비교 의미·정렬 알고리즘·반환 detail 구조·내부 타입(EngineState·Card 런타임 표현·ClozeBlank 등)은 **엔진규격.md 소관**이며 본 규격은 정의하지 않는다(경계 침범 금지).

```
Engine.init({ subjectId: string, config: SubjectConfig, decks: <deck.js 등록 객체 배열> }) -> <EngineState: 엔진규격 소관>
Grader.check({ input, card, normalize: string[], synonyms: Record<동의어,대표어> }) -> <엔진규격 소관 결과>
Leitner.next({ state, leitner: LeitnerConfig, now: epochMs }) -> <엔진규격 소관>
```

- 본 규격이 보장하는 것: `config`(§3.2 SubjectConfig)·`normalize`(§3.4 id 배열 string[])·`synonyms`(역인덱스 `Record<동의어,대표어>`)·`decks`(§2.3 deck 객체)·`card`(§2.4 Card)의 **형태**. 이들을 받아 동작하는 엔진의 내부·반환·정렬은 엔진규격이 정한다.
- **synonyms 변환 주체 = generate.py(_인터페이스계약 §8)**: config의 작성 형식 `{대표어:[동의어,...]}`(사람 친화)를 generate.py가 **역인덱스 `Record<동의어,대표어>`로 컴파일**해 deck/엔진에 공급한다. 엔진은 역인덱스만 소비(엔진은 대표어→배열 형식을 받지 않음).
- **자동채점 경계(불변)**: `self` grade_mode의 카드는 채점 보조 키가 없으므로(§2.4) 엔진이 기계비교 불가 → 사용자 O/X를 그대로 반영. 사용자 UI 입력 `o|x`(UI/카드 레이어)는 **앱(UI) 레이어가 `correct|incorrect`로 변환** 후 엔진에 전달하며, 엔진은 `correct|incorrect`만 수용(변환 주체=UI — _인터페이스계약 §5). exact/keyword/cloze는 기계비교 허용(헤더 용어정의 참조).
- **합격확률 반환 금지**: 본 규격이 넘기는 어떤 데이터도 확률 필드를 포함하지 않는다(불변경계).

### 4.3 type ↔ grade_mode 적합성 매트릭스 + 기본값(계약)

| type | 허용 grade_mode | 기본 grade_mode |
|---|---|---|
| `func` | exact, self | exact |
| `proc` | keyword, self | keyword |
| `recall_seq` | exact, self | exact |
| `cloze` | cloze, self | cloze |
| `judge` | exact, self | exact |

- _인터페이스계약 §4 정렬: `recall_seq`=exact(순서포함 배열 완전일치), `judge`=exact. 기존 sequence→keyword, judgment→self 매핑 폐기.
- `recall_seq`의 exact 채점은 step별 순서포함 원소단위 완전일치(엔진규격 소관). generate.py는 authoring `answers`(`\|` 구분, 순서 보존)를 `answer_spec.sequence: string[]`로 조립한다(§2.4.1).
- 허용 외 조합(카드 명시 grade_mode 또는 config 오버라이드) → `E_GRADE_TYPE_MISMATCH`.
- 이 매트릭스가 `answer_spec` 필수성의 단일 기준이다(§2.4: grade_mode 기준). `type:cloze`+`grade_mode:self`처럼 허용 조합이어도 grade_mode=self이면 `answer_spec`을 생략한다(self는 비교 안 함).

## 5. export / 백업 포맷

### 5.1 export 책임 경계
- **export(내보내기)**: 런타임 앱이 localStorage 상태를 §5.2 JSON 파일로 저장(서버0, 더블클릭). 트리거·파일명은 엔진/UI 소관.
- **import(복원)**: localStorage에 되쓰는 실제 복원은 **런타임 앱만** 수행한다(서버0). generate.py의 `import` 서브커맨드는 **복원이 아니라 백업 파일의 검증·리포트(드라이런)** 다 — localStorage에 쓰지 않는다(혼동 방지를 위해 §7 CLI는 `verify-backup`로 명명).

### 5.2 백업 JSON 스키마(계약)

```json
{
  "format": "study-backup",
  "schema": 1,
  "exported_at": "<ISO8601 UTC>",
  "subject_id": "<id>",
  "generator_version": "<semver>",
  "progress": {
    "<card_id>": {
      "box": <int 1..boxes>,
      "due": <epochMs>,
      "last_result": "correct" | "incorrect" | "skip",
      "cold_attempts": <int>,
      "cold_correct": <int>,
      "history": [ { "ts": <epochMs>, "result": "correct"|"incorrect"|"skip", "warm": <bool> } ]
    }
  },
  "meta": { "card_count": <int>, "schema_of_runtime": <int> }
}
```

- 키는 `card_id`(안정 id, deck namespace 내 유일 — §0) — content-hash 금지로 오탈자 수정 후에도 이력 매칭.
- verdict 어휘 = `correct | incorrect | skip` 3값으로 통일(_인터페이스계약 §5: `wrong` 폐기, 전 계약·localStorage·백업 동일).
- `warm`: 세션내 재큐 시도면 true(박스이동 무관, 기획 §3). 승급은 cold 첫시도 정답만.

### 5.3 import 검증(`verify-backup` 드라이런 — 복원 안전)
- `format != "study-backup"` → `E_BACKUP_FORMAT`.
- `schema` 미지원(코드가 모르는 버전) → `E_BACKUP_SCHEMA_UNSUPPORTED`. 지원 버전 범위 = `[1, SCHEMA_MAX]`(현재 SCHEMA_MAX=1). 하위 버전은 엔진규격 localStorage 마이그레이션 shim과 **동일 버전축**을 따른다(실제 shim 적용은 복원 주체인 런타임 앱 소관 — generate.py는 "마이그레이션 가능 여부"만 판정·리포트).
- 백업에만 있고 현 deck에 없는 `card_id` → **경고** `W_ORPHAN_PROGRESS`(보존, 폐기 금지 — 오답이력 유지 원칙). card_id 유일성 범위는 deck namespace(subject_id) 내(§0)이므로 매칭은 동일 subject 내 `card_id` 기준이다. 같은 subject의 다른 deck으로 "이동"해도 동일 card_id면 매칭 유지(이동 = 동일 id 보존이 원칙). 해당 subject 빌드의 전체 deck에서 card_id를 못 찾을 때만 orphan 판정.
- 백업 `box` 값이 현 config `leitner.boxes` 범위를 초과(박스 수 축소) → **경고** `W_BACKUP_BOX_OUT_OF_RANGE`(해당 항목 box를 `boxes`로 클램프 권고, 데이터 폐기 금지). 실제 클램프는 복원 주체 소관.

## 6. 에러 규약(generate.py & 빌드)

- 출력: stderr에 `<LEVEL> <CODE> <loc> <message>` 1줄/건. UTF-8 파일로도 동시 기록(`생성물/build_report.txt` 기본, `--report <path>`로 변경 가능 — Windows cp949 콘솔 회피).
- `<loc>` 표기 규칙: `deck_id:card_id`. deck/card 미상 컨텍스트는 빠진 자리를 `-`로 채움(예: config 에러 `-:-`, front-matter 파싱 실패 `<deck_id>:-`). loc 필드는 항상 존재(빈칸 금지).
- 종료코드: 에러(E_) 1건 이상 = exit 2 / 경고만 = exit 0 / 정상 = exit 0. (`--strict` 시 경고도 exit 2.)

| code | level | 조건 |
|---|---|---|
| `E_CARD_ID_DUP` | ERROR | card_id deck namespace(subject_id) 내 중복 |
| `E_CARD_ID_INVALID` | ERROR | `## @id` 마커 id가 §0 정규식 위반 |
| `E_DECK_ID_DUP` | ERROR | deck_id 과목내 중복 |
| `E_DECK_ID_CASE_COLLISION` | ERROR | deck_id 소문자 접기 충돌(§0.2) |
| `E_AREA_UNKNOWN` | ERROR | area ∉ {written,practical} 또는 subarea가 area별 enum 밖 |

## 7. generate.py 인터페이스(계약)

- 무의존 stdlib만(`argparse`, `json`, `re`, `pathlib`, `datetime`, `sys` 등). **content-id 생성용 `hashlib`(content-hash) 금지**(card_id는 작성자 명시). **3rd-party 금지**(YAML 파서 포함 — front-matter는 자작 파서, §1.2).
- CLI:

```
python generate.py build  --subject <subject_id> [--out <dir>] [--strict] [--clean] [--reproducible] [--report <path>]
python generate.py verify --subject <subject_id>            # 빌드 없이 검증만, exit 2 on E_
python generate.py verify-backup --backup <path> --subject <subject_id>   # 백업 검증·리포트(드라이런, 복원 안 함)
```

- `--strict`: 경고(W_)도 exit 2로 승격.
- `--clean`: 스테일 deck.js 삭제(미지정 시 경고만, §2.2).
- `--reproducible`: `generated_at` 고정값 출력(§2.5).
- **내부 함수 분해는 구현 자유**(함수명·시그니처 강제 안 함). 본 규격이 계약하는 것은 §8 테스트 대상(관찰가능 입출력)뿐이다.

## 8. 검증 게이트(④ 연동)

기획 §5: 채점기·정규화·cloze파서·Leitner는 **테스트셋 동반 필수**. 본 규격이 계약하는 테스트 대상(관찰가능 입출력 골든셋):
- **정규화 골든셋** — normalize id 배열(§3.4) 순서대로 적용한 입력→출력. synonyms 치환 위치(lower 직후) 케이스 포함. 배열 순서대로 적용한 **최종 입출력 동치**를 검증(§3.4).
- **cloze 토큰 파싱 + answer_spec 조립** — 토큰 0/1/다중·별칭·리터럴 중괄호 이스케이프(`\{`)·빈 토큰(에러) 케이스. 산출 `answer_spec.blanks`(§2.4.1) 구조 및 type별 answer_spec 필드(accepted/requiredKeywords/sequence) 검증.
- **md 파싱** — 정상 + 각 E_ 트리거 1케이스 이상(front-matter·분리마커·메타·정답키 누락 등).
- **deck.js/manifest.js 산출물 검증(stdlib-only, Node 없이)** — 산출 .js에서 등록 객체 리터럴(`= { ... };`)을 **구조 정규식으로 추출**해 `json.loads` 가능한 부분을 파싱하고, 원본 빌드 객체와 round-trip 동일성 검증. (전체 JS 실행이 아니라, generate.py가 자기 산출 객체를 json으로도 동시 산출해 두고 .js 추출본과 대조하는 방식 — Node 불요. §10 O2 확정.)

## 10. [OPEN] 잔여 (cross-contract 합의 — escalate 아님, 산출물 형태 조율)

> 아래는 본 규격 단독으로 계약을 닫았으나, **타 규격(개념서·엔진)과 산출물 위치/형태를 맞출 여지**만 남은 항목이다. 기계검증 계약 자체는 본 문서에서 확정됨.

- O1: 개념서 인덱스(section_id 목록) 제공 경로·포맷 — 개념서규격 의존. 본 규격은 "부재 시 스킵+경고"(§1.4)로 동작 확정. index.html 주입 방식(전담/치환)도 여기 묶임(§2.6).
- O2: deck.js/manifest.js 구조검증을 §8 "정규식 추출 + json round-trip"으로 확정. 엔진규격이 다른 검증을 요구하면 조율.

> 확정 완료(기존 OPEN 해소): cloze 이스케이프(§1.3) · intervals_days↔boxes 1대1(§3.6) · export/import 주체 분리(§5.1) · 재현빌드(§2.5) · due-first→weight 정렬은 엔진규격 소관으로 위임(본 규격은 정렬 명시 안 함, §4.2) · synonyms 치환 위치 lower 직후(§3.4) · 채점 보조키 grade_mode 기준 통일(§2.4) · card_id/grade_mode/blanks string[][] SoT 정렬(_인터페이스계약 §14).
