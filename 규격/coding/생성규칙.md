# coding 플러그인 생성규칙 (비용코어)

> buildflow cycle-2 ②명세. **런타임규격.md + 플러그인계약.md conform**.
> 목적: `LearningProfile`(깊이스무고개 STEP 2 산출) → `code-problem` ActivitySpec 배열을 *빡빡한 템플릿*으로 생성(추론 최소).
> 원칙: 사실·알고리즘은 **웹그라운딩 의무**(환각 코드·틀린 알고리즘 방지). 추론·판단 영역 최소화.

---

## 0. 전제 — 비용 원칙

생성 비용 구조:
```
총비용 ≈ (웹조사) + (템플릿 채우기 × 문제수)
```
- 웹조사는 STEP 0에서 이미 완료(`LearningProfile.subject_structure` 포함). 재조사 금지.
- 템플릿 채우기 = slot 삽입만. 자유 생성 금지.
- **환각 방지 규칙**: 알고리즘·함수명·라이브러리·수식이 등장하면 반드시 웹조사 출처(`grounding_url`) 첨부. 모델 기억 단독 사용 금지.

---

## 1. 입력: LearningProfile

깊이스무고개 STEP 2가 생성하는 프로파일(런타임규격 소비 계약과 독립된 생성 전용 입력).

```ts
interface LearningProfile {
  subject:           string;
  subject_structure: {
    level_system:    string[];      // 입문→숙련 단계 레이블
    topic_taxonomy:  string[];      // 하위 토픽 분류 (STEP 0 웹조사)
    eval_criteria:   string | null;
    common_traps:    string[];
    source_urls:     string[];      // STEP 0 조사 출처
  };
  estimated_level:   string;        // 진단 기반 추정 ("입문", "중급" 등)
  goal:              "exam" | "project" | "literacy" | "skill";
  scope:             string[];      // 대상 토픽 (subject_structure.topic_taxonomy 서브셋)
  depth_ratio:       { theory: number; practice: number };  // 합=1
  volume:            string;        // "30문제", "1시간분량" 등
  target_plugins:    string[];      // "coding" 포함 여부 확인
}
```

**전제 검사**: `target_plugins.includes("coding")` 가 false면 생성 스킵.

## 2. 생성 파이프라인 (4단계)

```
[STAGE A] 문제셋 계획 수립      → ProblemPlan[]
[STAGE B] 웹그라운딩            → GroundedContent (per plan)
[STAGE C] 템플릿 채우기         → ActivitySpec (draft)
[STAGE D] test_case 자동검증    → ActivitySpec (validated)
```

각 Stage는 순차. Stage B·C는 문제 단위 반복. Stage D는 전수 검사.

---

## 3. STAGE A — 문제셋 계획 수립

### 3-1. 문제 수 결정

| volume 표현 | 목표 문제 수 |
|---|---|
| "~30분" 이하 | 5문제 |
| "~1시간" | 10문제 |
| "N문제" 명시 | N |
| "전체 범위" | scope.length × 3 (최대 30) |

### 3-2. 난이도 분포 결정

`estimated_level` → 분포 자동 결정(사람 판단 불요):

| estimated_level | D1(기초) | D2(응용) | D3(심화) |
|---|---|---|---|
| 입문 | 60% | 35% | 5% |
| 초급 | 40% | 45% | 15% |
| 중급 | 20% | 50% | 30% |
| 고급 | 10% | 40% | 50% |

난이도 정의:

| 레벨 | 기준 | 예시 |
|---|---|---|
| D1 기초 | 단일 개념, 조건문/반복문/기본 I/O, 코드 5~15줄 | 홀짝 판별, 합계 |
| D2 응용 | 2개 이상 개념 결합, 리스트/딕셔너리 활용, 15~30줄 | 빈도 세기, 이진탐색 |
| D3 심화 | 알고리즘 설계, 재귀·DP·자료구조, 30줄 이상 | 최단경로, 트리 순회 |

### 3-3. 토픽 배분

`scope` 토픽을 문제 수에 고르게 배분. 나머지는 앞 토픽부터 +1씩.

```
base = floor(총문제수 / scope.length)
remainder = 총문제수 % scope.length
각 토픽별 문제 수 = base (앞 remainder개 토픽은 base+1)
```

### 3-4. ProblemPlan 형태 (출력)

```jsonc
// ProblemPlan[]
[
  {
    "plan_id":    "plan-001",           // 내부 식별자
    "topic":      string,               // scope 토픽 중 하나
    "difficulty": "D1" | "D2" | "D3",
    "concept":    string,               // 1줄 핵심 개념 설명 (한국어)
    "grounding_required": boolean,      // true이면 Stage B 웹조사 필수
    "tags": {
      "area":    string,
      "subarea": string,
      "unit":    string
    }
  }
]
```

**grounding_required 판단 규칙**:
- 알고리즘명(이진탐색·DFS·DP 등) 포함 → true
- 표준라이브러리 함수 사용 필요 → true
- 순수 제어흐름(조건문·반복문)만 → false

## 4. STAGE B — 웹그라운딩

### 4-1. 조사 의무 조건

`plan.grounding_required === true` 이면 **반드시** 웹조사 실행.

조사 내용:
1. 해당 알고리즘·개념의 표준 Python 구현 패턴
2. 사용할 표준라이브러리 함수 시그니처 + 동작 확인
3. 흔한 엣지케이스 (0, 음수, 빈 입력 등)

### 4-2. GroundedContent 형태

```ts
interface GroundedContent {
  plan_id:           string;
  reference_impl:    string;    // 조사로 확인한 정답 구현(Python)
  edge_cases:        string[];  // ["n=0일 때 0 반환", "빈 리스트 → 0" …]
  grounding_urls:    string[];  // 출처 URL
  verified:          boolean;   // 구현 정확성 확인 여부
}
```

`grounding_required === false`이면 `reference_impl = ""`, `grounding_urls = []`, `verified = true`(자명한 기초 로직).

## 5. STAGE C — 템플릿 채우기

### 5-1. ActivitySpec 생성 템플릿

빈칸(⟨…⟩) 채우기만 허용. 자유 생성 금지.

```
activity_id  = "cod-py-{plan_id 뒤 숫자}-{seq 3자리}"
plugin_id    = "coding"
type         = "code-problem"
weight       = D1→2 / D2→5 / D3→8
tags         = plan.tags (그대로)
enabled      = true
language     = "python"
```

**front.prompt 템플릿**:

```
⟨입력 조건⟩을 입력받아 ⟨출력 조건⟩을 출력하시오.
(⟨제약: 범위·형식·예외처리⟩)
```

규칙:
- 1~3 문장. 마지막 문장에 반드시 입력 형식과 출력 형식 명시.
- 예: `"정수 n (1 ≤ n ≤ 100)을 입력받아 1부터 n까지의 합을 출력하시오."`
- 애매한 표현 금지: "적절히", "올바르게", "잘" → 구체적 조건으로 대체.
- 한국어 작성.

**front.starter_code 규칙**:

| 난이도 | 스캐폴딩 수준 | 예시 |
|---|---|---|
| D1 | `input()` 호출 + `# 여기에 작성` | `n = int(input())\n# 여기에 작성\n` |
| D2 | `input()` 호출 + 함수 시그니처 제공 | `def solve(n):\n    # 여기에 작성\n    pass\nn = int(input())\nprint(solve(n))\n` |
| D3 | import + 함수 시그니처 + docstring | `import sys\ndef solve(graph, start):\n    """..."""\n    pass\n` |

규칙:
- starter_code는 반드시 실행 가능한 Python. SyntaxError 있으면 생성 오류.
- `print` 호출은 포함하되, 정답 로직은 비워둠.
- 주석은 한국어.

**back.solution**:
- `grounded_content.reference_impl` 기반으로 작성.
- `grounding_required=false`이면 직접 작성(단순 로직이므로 환각 위험 낮음).
- 반드시 모든 test_case 통과 확인(Stage D).

**back.explanation**:
- 3줄 이하. 핵심 알고리즘 아이디어 + 시간복잡도(D2 이상).
- 형식: `"핵심: … / 복잡도: O(…) / 주의: …"`

### 5-2. grading.test_cases 작성 규칙

**최소 케이스 수**:

| 난이도 | 필수 케이스 수 |
|---|---|
| D1 | 3개 이상 |
| D2 | 5개 이상 |
| D3 | 7개 이상 |

**케이스 구성 의무**:

| 카테고리 | 설명 | D1 | D2 | D3 |
|---|---|---|---|---|
| 기본 | 일반적인 입력 | 필수 | 필수 | 필수 |
| 경계 최솟값 | n=0, 빈 리스트, n=1 등 | 필수 | 필수 | 필수 |
| 경계 최댓값 | 문제 상한값 | - | 필수 | 필수 |
| 엣지케이스 | grounded edge_cases 반영 | - | 필수 | 필수 |
| 대용량 | 성능 검증 | - | - | 필수 |

**input 형식 규칙**:
- stdin 그대로: `"5\n"` (개행 포함), 여러 줄이면 `"3\n1 2 3\n"`.
- `input()` 호출 횟수와 입력 줄 수 반드시 일치.
- 공백 포함 시 split 방식 명시(prompt에서 "공백으로 구분" 등).

**expected 형식 규칙**:
- `print()` 출력 그대로(`stdout-trim` 비교이므로 앞뒤 공백·개행 무시됨).
- 부동소수 출력 문제는 소수점 자릿수 고정(예: `"3.14"`, `"3.14159"`). 절대 `"3.141592653..."` 형태 금지.
- 여러 줄 출력이면 `"1\n2\n3"` 형태(trailing newline 없어도 trim으로 처리됨).

## 6. STAGE D — test_case 자동검증

### 6-1. 검증 방법

Stage C에서 생성한 `back.solution`을 각 test_case에 대해 실행(Pyodide 또는 로컬 Python):

```
for each activity in generated_activities:
  for each tc in activity.grading.test_cases:
    actual = run_python(activity.back.solution, stdin=tc.input)
    assert trim(actual) == trim(tc.expected), f"[{activity.activity_id}] tc fail: input={tc.input!r}"
```

### 6-2. 검증 실패 시 처리

| 실패 유형 | 처리 |
|---|---|
| solution 오류(예외 발생) | Stage B 재조사 → solution 재작성 → 재검증 |
| expected 불일치 | expected 값 수정 OR test_case 재설계 |
| 타임아웃(5s 초과) | D3는 test_case 입력 크기 축소, 또는 알고리즘 교체 |
| 재검증 2회 이상 실패 | 해당 plan을 `[FAILED]` 마킹, 생성물에서 제외 + 사용자 보고 |

### 6-3. 검증 결과 메타데이터

각 ActivitySpec에 추가(생성물 전달 전 제거 가능):

```jsonc
"_meta": {
  "plan_id":          string,
  "grounding_urls":   string[],
  "validation_pass":  boolean,
  "generated_at":     "ISO8601"
}
```

## 8. 난이도별 생성 가이드라인 (템플릿 보조)

### D1 기초 — 전형적 문제 패턴

| 패턴 | prompt 템플릿 조각 | test_case 수 |
|---|---|---|
| 합계/평균 | "n개의 정수를 입력받아 합계를 출력" | 3 |
| 조건 분기 | "정수를 입력받아 …이면 A, 아니면 B 출력" | 3 (양쪽+경계) |
| 반복 출력 | "n을 입력받아 1부터 n까지 출력" | 3 |
| 문자열 처리 | "문자열을 입력받아 … 출력" | 3 |

### D2 응용 — 전형적 문제 패턴

| 패턴 | grounding 대상 | test_case 수 |
|---|---|---|
| 정렬 응용 | `sorted()`, `list.sort()` 시그니처 확인 | 5 |
| 딕셔너리 빈도 | `collections.Counter` 또는 dict | 5 |
| 이진탐색 | bisect 또는 직접 구현 — 웹조사 필수 | 5 |
| 스택/큐 | `collections.deque` 확인 | 5 |

### D3 심화 — 전형적 문제 패턴

| 패턴 | grounding 대상 | test_case 수 |
|---|---|---|
| DP(최적화) | 점화식 정확성 — 웹조사 필수 | 7 |
| 그래프(BFS/DFS) | 인접리스트·방문배열 표준 구현 | 7 |
| 재귀+메모이제이션 | lru_cache 사용 여부 확인 | 7 |
| 문자열 알고리즘 | KMP·슬라이딩윈도 등 — 웹조사 필수 | 8 |
