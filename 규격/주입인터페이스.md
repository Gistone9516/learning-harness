# v5 주입 인터페이스 규격

> buildflow ② per-folder 계약. **`_인터페이스계약.md`(SoT)에 conform.** 충돌 시 SoT 권위.
> 범위: 소비 프로젝트가 **콘텐츠·config를 어떻게 제공하나**(프레임워크↔소비 경계). 이 리포엔 실제 과목 없음(목업만).
> 형식 = JSON 파일(코드 아님, 데이터). md→JSON 빌드 헬퍼는 선택(백로그), 계약은 JSON 형태.

---

## 1. 마운트 콘텐츠 폴더 구조

소비 프로젝트의 콘텐츠 폴더(봇이 마운트, SoT §3 부트):
```
<mount>/
├ manifest.json              ← 과목·덱 레지스트리(§2)
├ decks/
│   └ <deck_namespace>.json  ← 덱별 카드(§3)
├ config/
│   └ <deck_namespace>.json  ← 과목 config: 정규화·synonyms·채점매핑·leitner·능력활성(§5,§7)
├ _상태/                     ← 봇이 생성(진도, SoT §2). 소비자가 두지 않음
└ (선택) src/                ← md→JSON 빌드 스크립트 등 소비자 도구(계약 무관)
```
- `deck_namespace`는 콜론 금지 문자열(한글 허용). manifest·decks·config의 키가 일치해야 함.

---

## 2. manifest.json

```jsonc
{
  "schema_version": 1,
  "subject": "<subject_id>",          // 과목 식별
  "decks": [
    {
      "namespace": "<deck_namespace>",
      "title": "표시명",
      "card_count": 42,               // 빌드 시점(검증용)
      "config_ref": "config/<deck_namespace>.json",  // 과목 config 경로(선택, 없으면 코어 기본)
      "built_at": 1718000000000       // epoch ms
    }
  ]
}
```
- 봇은 manifest를 읽어 사용 가능 deck를 안다. 부재 시 `ManifestMissingError`(SoT §6).

## 3. deck JSON (cards)

```jsonc
{
  "namespace": "<deck_namespace>",
  "cards": [ CardDef, ... ]            // CardDef = SoT §1
}
```
**CardDef authoring(JSON)** — SoT §1 필드. 중첩 구조 강제(평면 직렬화 금지):
```jsonc
{
  "card_id": "nlp-lda-define-01",     // ^[a-z][a-z0-9-]{2,63}$, 콜론 금지, 안정 id
  "schema_version": 1,
  "subject": "<subject_id>",
  "unit": "<unit_id>",
  "type": "func|proc|recall_seq|cloze|judge",
  "grade_mode": "exact|keyword|cloze|self",
  "front": { /* type별 §4 */ },
  "back":  { "detail": "...", "note": "...", "why": "..." },   // 전부 선택
  "answer_spec": { /* §4, self면 null */ },
  "tags": { "weight": 5, "area": "...", "subarea": "..." },     // weight [1,10] 기본 5, area/subarea 선택
  "links": { "concept_ref": "<ref_id>" },                      // 선택
  "enabled": true
}
```

---

## 4. 카드 authoring (type별 front / answer_spec)

| type | grade_mode(권장) | front 키 | answer_spec |
|---|---|---|---|
| `func` | exact 또는 self | `{prompt, hint?}` | `{accepted:[...], normalize:[...]}` |
| `proc` | keyword 또는 self | `{prompt, hint?}` | `{required_keywords:[[...],...], normalize:[...]}` |
| `recall_seq` | exact 또는 self | `{prompt}` | `{sequence:[...], normalize:[...]}` (순서포함 완전일치) |
| `cloze` | cloze 또는 self | `{text}`(마커 `{{0}}`,`{{1}}`…) | `{blanks:[[...],...], normalize:[...]}` (0-base) |
| `judge` | exact 또는 self | `{scenario, options:[...]}` | `{accepted:[정답옵션], normalize:[...]}` |

- **self 카드**: `answer_spec=null`(앱 정답 미보유, 사용자 자가판정). `grade_mode="self"`.
- 매핑 위반(예: cloze인데 blanks 없음)은 load-time 거부(§6).
- `front`/`back`의 추가 키(보충 설명·이미지 url 등)는 허용(봇이 무시 또는 렌더). 단 `answer_spec`은 평면 키 금지(객체).

## 5. SubjectConfig (config/<deck>.json)

과목 특화. 엔진 무오염 — 콘텐츠 어휘·간격은 전부 여기로 주입(SoT §1 LeitnerConfig·정규화·synonyms).
```jsonc
{
  "normalize_profiles": {              // type별 기본 정규화 규칙 id 배열(카드가 미지정 시 사용)
    "func":  ["nfkc","trim","collapse_space","lower"],
    "proc":  ["nfkc","trim","collapse_space","lower"],
    "cloze": ["nfkc","trim","strip_all_space","lower"]
  },
  "synonyms": {                        // 대표어 -> [동의어...] (엔진이 역인덱스로 컴파일, lower 직후 치환)
    "함수": ["function","펑션"]
  },
  "scoring_overrides": {               // (선택) 특정 card_id의 grade_mode 재정의
    "nlp-xx-01": "keyword"
  },
  "pass_targets": {                    // (선택) 합격경로 target. subarea -> 목표점수[정수]. 미지정 시 pass_path 집계 제외
    "<subarea>": 70
  },
  "leitner": {                         // (선택) LeitnerConfig. 미지정 시 코어 기본(SoT §0)
    "intervals_days": { "1": 1, "2": 3, "3": 7 },
    "dday_compress_days": 1
  }
}
```
- **scoring_overrides 적용**: 봇 부트가 `card_id → 실효 grade_mode` 맵으로 컴파일(CardDef.grade_mode에 override 반영), `Ctx.grade_mode_of`로 핸들러에 공급(봇계약 §3·§4). 엔진은 ScoreInput.mode만 본다.
- **pass_targets**: 엔진계약 §5 `get_dashboard_data(pass_targets=...)`로 주입. 과목 어휘·임계는 전부 여기서(엔진 하드코딩 금지, SoT §7.1). subarea가 "필기/실기" 같은 분류든 무관 — 점수 목표만 매핑.
- 정규화 규칙 id 카탈로그(9종)는 `엔진계약.md §3.4` 권위. config는 그 id만 나열.
- synonyms는 `{대표어:[동의어]}`로 작성 → 봇 부트가 `{동의어:대표어}` 역인덱스로 컴파일해 엔진에 공급(SoT §1 ScoreInput.synonyms).

## 6. 검증 (load-time, `ContentInjectionError`)

봇 부트(SoT §3 3단계)에서 주입 콘텐츠를 검증, 위반 시 `ContentInjectionError`로 **부트 차단**(오채점 방지):
- card_id 정규식 + deck 내 유일성. 콜론 금지(card_id·namespace).
- `subject`·`unit`·`enabled` 필수. `type`/`grade_mode` enum.
- type↔front 키 정합(예: cloze는 `text` + 마커, judge는 `options`).
- answer_spec 정합: exact→accepted≥1 / keyword→required_keywords≥1그룹 / cloze→blanks(빈칸수=마커수) / recall_seq→sequence≥1 / self→null.
- normalize 규칙 id가 카탈로그(9종)에 존재.
- manifest `card_count` vs 실제 카드 수(불일치=경고).
- config synonyms·normalize_profiles 형태.
- **1차 방어선은 빌드/주입 단계.** 런타임은 깨지지 않게 폴백(정적앱 무중단 우선) — 단 주입 검증 실패는 부트 차단(런타임 도달 전).

## 7. 능력 활성화 (어떤 capability를 켤지)

소비 프로젝트가 4계층 능력(SoT §4 레지스트리)에서 쓸 것을 config로 선택(빼기 미학 = 소비 앱):
```jsonc
{
  "capabilities": {
    "enabled": ["recall_self","mcq_buttons","cloze_modal","srs_due_alert","dashboard_live"],
    "ai": { "enabled": ["ai_openend_grade","ai_hint"], "model": "<저렴 모델 id, 예: haiku tier>", "effort": "low" }
  }
}
```
- `enabled` 미지정 시 core tier 전체 활성(기본). extension·ai는 명시적 opt-in.
- `capability_id`는 `학습타입규격.md` 권위. 알 수 없는 id = `ContentInjectionError`.

## 8. 목업 예시 (`_예시/`)

개발·검증용 최소 덱(실제 과목 아님). 얇은 슬라이스(기획 §10)용:
```
_예시/
├ manifest.json              // subject "demo", deck "demo-core" 1개
├ decks/demo-core.json       // func/cloze/judge/self 각 1~2장
└ config/demo-core.json      // 최소 정규화 프로파일
```
- 봇 회귀·스모크가 이 목업으로 전 루프(제시→입력→채점→Leitner→저장→재출제)를 증명.
