## 0. 용어·전역 상수

| 상수 | 값 | 설명 |
|---|---|---|
| `BOX_MIN` | `1` | 최소 박스 번호 |
| `BOX_MAX` | `3` | 최대 박스 번호(box3 도달 후 cold 정답 시 졸업 처리) |
| `BOX_INTERVALS_DAYS` | `{1:1, 2:3, 3:7}` | 박스→재출제 간격(일). 키=박스번호, 값=일수. **기본값**(config 주입으로 오버라이드 가능, §1.1·§4 무오염) |
| `SCHEMA_VERSION` | `1` | localStorage 스키마 현재 버전(정수, 단조증가) |
| `STORAGE_KEY_PREFIX` | `"clf:"` | localStorage 키 접두(clf = card learning framework). 마이그레이션 호환을 위해 고정 |
| `MS_PER_DAY` | `86400000` | 1일 밀리초 |
| `DDAY_COMPRESS_DAYS` | `1` | D-day 모드 간격압축 상한일(기본값, config 주입 가능 §2.4) |

- **시각 표현 규약**: 모든 저장 시각 = `epoch ms`(정수, `Date.now()` 기준, UTC). 날짜 비교는 ms 단위 직접 비교(타임존 변환 없음).
- **due 판정 경계**: `now >= due_at` → due(경계 포함, `>=`).
- **간격 상수의 위치**: `BOX_INTERVALS_DAYS`·`DDAY_COMPRESS_DAYS`는 코어 **기본값**일 뿐이며, 엔진은 이를 하드코딩된 의미로 해석하지 않고 `LeitnerConfig`(§생성규칙 §3.6, config 주입)가 주어지면 그 값을 사용한다. config 미주입 시 위 기본값을 쓴다(2번째 과목 무재협상 보장).

## 1. Leitner 코어

### 1.1 박스 정의
- 박스 = 정수 `box ∈ {BOX_MIN..BOX_MAX}`(기본 1..3). 신규/미학습 카드의 초기 박스 = `BOX_MIN`(1).
- 박스별 재출제 간격(일) = `BOX_INTERVALS_DAYS`(기본 box1→1일, box2→3일, box3→7일). 실제 값은 주입된 `LeitnerConfig.intervals_days`가 있으면 그것을 따른다.
- **졸업(graduated)**: 카드가 `box == BOX_MAX`인 상태에서 다시 **cold 정답**을 받으면 `graduated = true`로 설정한다(박스는 BOX_MAX 유지, due_at은 BOX_MAX 간격 후로 재설정). 졸업 카드는 due 도래 시 재검증용으로 계속 큐에 들어온다(영구 제외 아님). 졸업 조건은 "BOX_MAX **도달**"이 아니라 "BOX_MAX **에서 다시 cold 정답**"이다(§1.3과 정합).

### 1.2 시도(attempt) 분류 — 승급 규칙의 전제
- **cold attempt**: 해당 세션에서 그 card_id를 **처음** 푸는 시도. 세션당 card_id별 cold는 정확히 1회.
- **warm attempt**: 같은 세션 내에서 그 card_id를 **재큐로 다시** 푸는 시도(2번째 이후 모든 시도).
- 분류 판정 = 세션 휘발 집합 `session.seen_card_ids: Set<card_id>` 에 card_id 존재 여부.
  - 시도 직전 `seen_card_ids.has(card_id) == false` → cold, 그 후 add.
  - `true` → warm.
- **분류 시점**: attempt_kind는 카드가 사용자에게 제시되어 응답이 발생한 시점에 확정한다. `skip`(건너뜀)도 하나의 시도이며, cold/warm으로 분류되고 `seen_card_ids`에 add된다(아래 누적 규칙 §1.3 참조).

### 1.3 승급/강등 규칙 (계약)
입력: `(card_state, attempt_kind, verdict)` → 출력: `card_state'`(새 박스·due_at).

| attempt_kind | verdict | 박스 전이 | due_at 갱신 | graduated |
|---|---|---|---|---|
| `cold` | `correct` | `box → min(box+1, BOX_MAX)` | `now + interval[new_box]*MS_PER_DAY` | `box == BOX_MAX`였고 정답이면 `true` |
| `cold` | `incorrect` | `box → BOX_MIN`(강등) | `now + interval[BOX_MIN]*MS_PER_DAY` | 변화 없음 |
| `warm` | `correct` | **변화 없음**(박스무관) | **변화 없음** | 변화 없음 |
| `warm` | `incorrect` | **변화 없음**(박스무관) | **변화 없음** | 변화 없음 |
| `cold` | `skip` | **변화 없음** | **변화 없음** | 변화 없음 |
| `warm` | `skip` | **변화 없음** | **변화 없음** | 변화 없음 |

- **핵심 불변(v4 §3)**: 박스 이동은 **cold 첫시도 정답에만** 의존. 세션내 재큐(warm) 결과는 박스/간격에 **절대 영향 없음**.
- **이진 판정(부분점수 없음)**: `verdict ∈ {"correct", "incorrect", "skip"}` 3값뿐. **`partial`은 존재하지 않는다** — 모든 채점 모드는 정/오 이진이며(§3.1), 세션내 재큐가 교정장치다(v4 §3). 따라서 Leitner 전이도 `partial` 경로가 없다.
- **cold_attempts 누적 규칙**: cold attempt가 발생하면(verdict가 correct/incorrect/skip 무엇이든) `cold_attempts += 1`. `cold_correct`는 cold+correct일 때만 `+= 1`. **단, `skip`은 박스/due를 바꾸지 않지만 cold_attempts에는 카운트된다**(세션당 cold 1회 정의 유지 + 진도바 분모 일관). cold skip은 `cold_correct`를 증가시키지 않는다.
- **`graduated` 카드의 반복 전이**: 이미 `graduated == true`인 카드가 due 도래로 재출제되어 cold 정답이면 `box`는 BOX_MAX 유지, `due_at = now + interval[BOX_MAX]*MS_PER_DAY`로 재설정, `graduated`는 `true` 유지. cold 오답이면 일반 cold incorrect 규칙(box→BOX_MIN, `graduated → false`)을 따른다.

### 1.4 함수 시그니처
```ts
// 순수 함수, 부수효과 없음. now는 호출자가 주입(테스트 결정성).
// interval 값은 leitnerCfg(주입)에서 해소; 미주입 시 BOX_INTERVALS_DAYS 기본.
function leitnerTransition(
  state: CardProgress,
  attemptKind: "cold" | "warm",
  verdict: Verdict,
  now: number,            // epoch ms
  leitnerCfg?: LeitnerConfig
): CardProgress           // 새 상태(불변 복사 반환)

function nextDueAt(box: number, now: number, leitnerCfg?: LeitnerConfig): number  // now + interval[box]
function isDue(state: CardProgress, now: number): boolean  // now >= state.due_at
```
- `Verdict = "correct" | "incorrect" | "skip"`.

## 2. 출제큐 (selection / ordering)

### 2.1 큐 소스 구분
- `new` = 한 번도 cold 시도된 적 없는 카드(`cold_attempts == 0`). progress 맵에 항목이 없는 카드도 new로 본다(기본값 주입 시점 §2.6).
- `review` = `cold_attempts > 0` **이고** `isDue(state, now) == true` 인 카드. 졸업 카드(`graduated == true`)도 due면 review로 분류한다(§1.1).
- `new`와 `review`는 `cold_attempts == 0` 여부로 **상호배타** 분류된다. 신규카드의 기본 `due_at`은 `0`이라 `isDue`가 항상 true이지만, new는 `cold_attempts == 0` 조건으로 먼저 분류되므로 review와 겹치지 않는다.
- due 아닌 학습이력 카드(`cold_attempts>0` & not due) = 큐 제외(D-day 모드 예외 §2.4).

### 2.2 우선순위 규칙 (v4 §3 "출제 우선순위")
정렬 키(상위→하위):
1. **due-first**: `review`(due 카드) > `new`. (due 복습이 신규보다 항상 우선)
2. review 그룹 내부: `weight DESC`(빈출·배점 가중 높은 순).
3. new 그룹 내부: `weight DESC`(신규는 weight 우선 — v4 명시).
4. 동률 tie-break: **결정적 정렬**일 것(같은 입력 → 같은 순서). 기본 구현은 `card_id` 오름차순.

```ts
function buildQueue(
  cards: CardDef[],            // deck의 카드 정의
  progress: ProgressMap,       // card_id → CardProgress
  now: number,
  opts: QueueOptions
): card_id[]                    // 출제 순서 확정 배열
```
- **빈 결과**: 큐 대상이 없으면(전부 due 아님·신규 없음 등) 빈 배열 `[]` 반환. "학습할 카드 없음"은 호출측이 `length == 0`로 판정한다(별도 신호 없음).
- **고아/신규 progress**: `cards`에 있으나 `progress`에 없는 card_id는 new로 보고 §4.2 기본 CardProgress를 **메모리상에서 가정**하여 분류한다(저장은 첫 시도 시점에만). `progress`에 있으나 `cards`에 없는 고아 항목(덱 개정으로 카드 삭제)은 큐에서 제외하되 progress에서 삭제하지 않는다(오답이력 보존).

### 2.3 weight 정의
- `tags.weight: number`(CardDef의 `tags` 오브젝트 안 필드, SoT §14 §6). 허용 범위 `[1, 10]`. 기본값 `5`(미지정 카드).
- 의미 = 빈출도·배점 가중(생성규칙/콘텐츠가 부여). 엔진은 정렬에만 사용, 의미 해석 안 함.
- **입력값 정상화(런타임 관용)**: weight가 범위를 벗어나거나(`<1` 또는 `>10`) 비정수면 `[1,10]`으로 클램프 후 사용한다. 숫자가 아니거나 `NaN`이면 기본값 `5`를 적용한다. 정상화는 정렬 안정성 보장이 목적이며, 잘못된 weight는 빌드 단계(생성규칙 generate.py)에서 거부하는 것이 1차 방어선이다(런타임은 깨지지 않게 폴백만 — 정적앱 무중단 우선).

### 2.4 D-day 모드 (전박스 강제소환)
- 트리거: `opts.dDayMode == true` (UI/학습플랜이 D-day 임박 판정 시 주입; **엔진은 D-day 판정을 하지 않는다**). 트리거/해제 판정 주체는 호출측(UI/학습플랜).
- 동작:
  - **due 무시**: 모든 학습이력 카드(box 1/2/3 전부) + new 카드를 큐에 소환("막판 총정리").
  - **간격압축(정답)**: D-day 모드 중 cold 정답 승급 시 `due_at = now + min(원래간격, DDAY_COMPRESS_DAYS) * MS_PER_DAY`.
  - **오답(강등)**: D-day 모드 중 cold 오답도 일반 규칙(box→BOX_MIN, interval[BOX_MIN]=1일)을 그대로 따른다(압축 대상 아님 — 이미 최단간격).
  - 정렬: `box ASC`(약한 박스 먼저) → `weight DESC` → tie-break 결정적(기본 card_id ASC).
- `DDAY_COMPRESS_DAYS` 기본값 = `1`. config 주입 시 그 값을 사용.

### 2.5 QueueOptions
```ts
type QueueOptions = {
  dDayMode: boolean;           // 기본 false
  newCardLimit?: number;       // 세션당 신규 상한, 기본 무제한(undefined)
  reviewLimit?: number;        // 세션당 복습 상한, 기본 무제한
  deckNamespace: string;       // 대상 deck(§5)
}
```
- **limit 적용 순서**: 각 그룹(new/review)을 §2.2 키로 **먼저 정렬한 뒤** 그룹별로 상한만큼 자른다(상위 우선순위가 살아남음). review를 review limit로, new를 new limit로 독립 적용한 후 §2.2 순서(review 먼저)로 이어붙인다. limit 초과분의 다음 세션 이월은 하지 않는다(세션은 휘발, 다음 세션에서 다시 정렬·선발).

## 3. 채점코어 인터페이스 (입력 → 판정 → 피드백)

> v4 §2: **자동채점 없음**(실기 손풀이 측정 안 함). 본 채점코어는 **카드퀴즈 인출 응답**의 텍스트/선택 판정에 한정. 실기 procedural 측정 아님.

### 3.1 채점 4모드 (v4 §3) — 모두 이진 판정
`ScoreMode = "exact" | "keyword" | "cloze" | "self"`

| mode | 입력 | 판정 로직 | 결과 |
|---|---|---|---|
| `exact` | 사용자 문자열 | 정규화 후 정답집합(`accepted[]`)과 완전일치(복수정답 OR) | correct / incorrect |
| `keyword` | 사용자 문자열 | 정규화 후 **필수 키워드(`requiredKeywords`) 전부 포함** → correct, 하나라도 누락 → incorrect | correct / incorrect |
| `cloze` | 빈칸별 문자열 배열 | 각 빈칸 정규화 후 후보(OR)와 일치; **모든 빈칸 일치 → correct, 하나라도 불일치 → incorrect** | correct / incorrect |
| `self` | O/X 자가판정 | 사용자가 보낸 `selfVerdict` 그대로 채택(앱은 정답 미보유) | correct / incorrect |

- **부분점수 없음(v4 §3)**: keyword·cloze는 일부만 맞아도 **incorrect**다. 부분충족·가점·부분승급 개념이 없다. `matched`/`missed`는 **피드백 표시용**일 뿐 판정에 영향 없다.
- **keyword 가점 그룹 없음**: 모든 `requiredKeywords`는 필수다. "가점용/선택 키워드"(required:false) 개념은 v1에서 제거한다(이진 판정만).

### 3.2 입력 계약
```ts
type ScoreInput = {
  mode: ScoreMode;
  userAnswer: string | string[] | ("correct"|"incorrect");  // mode별
  answerSpec: AnswerSpec;        // 카드 정의에서 주입(정답·키워드·빈칸·normalize 포함, SoT §14)
  synonyms?: Record<string,string>;  // 동의어→대표어 역인덱스(런타임 형식, SoT §8). generate.py가 config의 {대표어:[동의어]}를 컴파일해 공급
  // normalize는 answerSpec.normalize 경유(ScoreInput 최상위 아님, SoT §14 §7)
}
```
- mode↔userAnswer 타입 매핑(위반 시 throw `ScoreInputError`):
  - exact → `string`
  - keyword → `string`
  - cloze → `string[]`(빈칸 수 = `answerSpec.blanks.length`, 불일치 시 throw `ScoreInputError`; 빈칸 인덱스는 0-base — 마커 `{{0}}`,`{{1}}`…와 정합, SoT §10)
  - self → `"correct" | "incorrect"`(그 외 값은 throw `ScoreInputError`)
- **빈 입력 처리**: 사용자답이 빈 문자열(정규화 후 `""`)이면 정답후보와 일치하지 않는 한 `incorrect`. cloze 배열의 특정 원소가 `""`면 그 빈칸은 불일치로 처리(해당 빈칸 후보에 `""`가 명시 포함되지 않는 한). 빈 입력은 throw가 아니라 `incorrect`다.
- **skip은 채점 입력이 아니다**: 사용자가 건너뛰면 호출측이 `score()`를 호출하지 않고 직접 `verdict = "skip"`을 Leitner/저장에 전달한다(§3.5의 verdict 영역과 정합). `score()`는 `correct`/`incorrect`만 반환한다.

### 3.3 AnswerSpec
```ts
type AnswerSpec = {
  // exact: 복수정답 허용
  accepted?: string[];          // exact 정답 후보(OR), 최소 1
  // keyword: 필수 키워드 집합(전부 포함 필수)
  requiredKeywords?: string[][];  // keyword 모드. 내부배열=한 필수그룹의 동의어 후보(any-of), 모든 그룹 필수. 최소 1그룹(SoT §14)
  // cloze: 빈칸별 정답(빈칸마다 복수정답 OR), 빈칸 인덱스 0-base
  blanks?: string[][];          // cloze, blanks[i] = i번 빈칸 정답후보(각 최소 1), i는 0-base(SoT §10)
  // recall_seq: 순서 있는 단계 배열(순서 의미)
  sequence?: string[];          // recall_seq 정답 단계 배열(순서 의미), 각 단계 최소 1, 최소 1단계(SoT §9)
  // 정규화 규칙 id 배열(SoT §14 §7): answer_spec 내부 필드. ScoreInput 최상위 아님.
  normalize: string[];          // 이 카드에 적용할 정규화 규칙 id 순서배열(§7). 미지정 시 과목 config 기본 프로파일
  // self: 정답 없음(앱 미보유) → AnswerSpec 비움 허용(normalize 포함 생략 가능)
}
```
- `accepted`/`requiredKeywords`/`blanks[i]`/`sequence`는 비어있지 않은 배열이어야 한다(빈 배열은 빌드단계에서 거부; 런타임 도달 시 해당 모드는 일치 불가로 `incorrect`). 빈 배열과 미지정의 의미 차이는 두지 않는다(둘 다 "후보 없음").
- **recall_seq 순서 채점(SoT §9)**: 카드 type `recall_seq`의 정답 = `sequence: string[]`(순서 의미). exact 채점(recall_seq)은 사용자 단계 배열을 step별 정규화 후 `sequence`와 **순서포함 원소단위 완전일치**로 판정한다. 부분점수 없음(이진: correct/incorrect). 길이 불일치는 incorrect.

### 3.4 정규화 모델 (규칙 id 배열 — 과목 config 주입, 엔진 무오염)
- 정규화 모델 = **순서 있는 규칙 id 배열** `normalize: string[]`(SoT §7). 배열 **순서대로** 적용한다(= 계약, 결정성). boolean 플래그 객체(구 `NormalizeConfig`)는 **폐기**.
- **공유 카탈로그(규칙 id → 동작)**. 카탈로그 정의의 SoT는 카드규격 §3.5이며, 엔진은 아래 동작 의미로 각 id를 해소한다:

| 규칙 id | 동작 |
|---|---|
| `nfkc` | 유니코드 NFKC 정규화(전각/합자/호환문자 통일) |
| `trim` | 문자열 앞뒤 공백 제거 |
| `collapse_space` | 연속 공백류(스페이스·탭·개행·전각공백·NBSP 포함, 모든 유니코드 공백)를 단일 스페이스로 축약 |
| `strip_all_space` | 모든 공백류 제거(축약 아닌 완전 삭제) |
| `lower` | ASCII 영문 소문자화 |
| `fullwidth_to_halfwidth` | 전각 영숫자·기호를 반각으로 변환 |
| `unify_cell_dollar` | 셀 참조의 `$` 표기 통일 |
| `unify_arg_sep` | 인자 구분자 표기 통일 |
| `strip_trailing_paren` | 후행 괄호 제거 |

- 정규화는 **양변(사용자답·정답)에 동일 적용**. `normalize` 배열에 없는 규칙은 적용하지 않는다(미지정 단계 건너뜀).
- 카드 authoring `grading.normalize`(id 리스트)는 generate.py가 `answerSpec.normalize`로 조립한다. 엔진은 `answerSpec.normalize`를 수용한다(ScoreInput 최상위 normalize 없음, SoT §14). 미지정 시 과목 config의 type별 기본 프로파일(id 리스트)을 사용한다.
- **`synonyms` 치환 위치(계약)**: `lower` **직후·비교 직전**에 적용한다(카드규격·생성규칙과 동일 위치 계약, SoT §8). `synonyms`는 `ScoreInput.synonyms`(런타임 역인덱스 `Record<동의어,대표어>`)로 주입된다. 공백으로 토큰 분리 후 각 토큰이 `synonyms`의 키(=동의어)이면 값(=대표어)으로 치환, 다시 단일 스페이스로 결합한다. 토큰 매칭은 정확일치(부분문자열 아님). 다어절(구) 동의어는 v1 범위 밖(토큰 단위만). `synonyms` 미지정/빈 객체면 치환 없음.
- **전각→반각 중복 회피**: `nfkc`와 `fullwidth_to_halfwidth`를 함께 둘 경우 작성자(생성규칙)가 중복되지 않게 프로파일을 구성한다. 엔진은 배열에 명시된 id만 순서대로 적용할 뿐 임의로 추가하지 않는다.
- **synonyms와 exact 상호작용**: exact도 위 정규화 파이프라인을 양변에 적용하므로 단일 토큰 정답도 synonyms 치환을 받는다(토큰 분리 결과가 1개여도 동일 규칙).

### 3.5 출력(판정 + 피드백)
```ts
type ScoreResult = {
  verdict: "correct" | "incorrect";   // score()는 이 둘만 반환(skip은 호출측이 직접 생성)
  matched: string[];          // 맞은 키워드/빈칸후보(피드백용), exact 정답이면 [일치한 정답] else []
  missed: string[];           // 누락 키워드 / 불일치 빈칸 라벨(피드백용)
  normalizedUser: string | string[];   // 정규화 후 사용자답(디버그·피드백, 입력형태 보존)
  feedback: FeedbackPayload;  // UI 표시용(엔진은 데이터 생성만, 렌더 아님)
}
type FeedbackPayload = {
  highlightMissed: string[];   // 강조할 누락 항목(keyword 누락어 / cloze 불일치 빈칸 인덱스 문자열)
}
type Verdict = "correct" | "incorrect" | "skip";   // 시스템 전체 verdict(Leitner/저장용). score()는 skip 미생성.
```
- **`showBack` 제거**: 뒷면(상세)은 모든 응답(correct/incorrect/skip)에서 노출되므로 분기 플래그가 불필요하다. 노출 시점은 UI규격 소관(엔진은 ScoreResult만 반환).
- **cloze matched/missed 의미**: `matched`/`missed`/`highlightMissed`의 cloze 원소는 **빈칸 인덱스(0-based)를 문자열화한 값**("0","1",…)이다. 빈칸이 `blanks[i]` 후보 중 하나와 일치하면 matched, 아니면 missed. 어떤 후보 문자열이 일치했는지는 반환하지 않는다(인덱스만).

### 3.6 함수 시그니처
```ts
function score(input: ScoreInput): ScoreResult   // 순수함수, 부수효과 없음
function normalize(s: string, rules: string[], synonyms?: Record<string,string>): string   // rules 배열 순서로 결정적 적용(§3.4), synonyms는 lower 직후 치환
```
- **불변**: `score`는 절대 정답률·합격확률을 산출하지 않는다(v4 경계). verdict(correct/incorrect) + 피드백만.
- self 모드: `score`는 `userAnswer`(correct|incorrect)를 그대로 verdict로 반환(검증 없음, 앱 정답 미보유).
- 결정성: 같은 `(input)` → 같은 `ScoreResult`.

## 4. localStorage 스키마 (버전필드 + 마이그레이션 shim)

### 4.1 키 네임스페이스 규약
- 진도 데이터 키: `STORAGE_KEY_PREFIX + deckNamespace + ":progress"` → 값 = `ProgressStore`(JSON).
- 메타 키: `STORAGE_KEY_PREFIX + "meta"`(= `"clf:meta"`) → 값 = `MetaStore`(전역, deck 무관).
- 세션 키: **저장 안 함**(세션은 메모리 휘발, 새로고침 시 리셋). `seen_card_ids` 등 세션 상태는 영속화하지 않는다(v1; 새로고침 = 세션 재시작).
- **card_id가 1차 키**(v4 §5): `ProgressStore.cards`는 `card_id → CardProgress` 맵.
- **키 구성문자 제약**: `deckNamespace`와 `card_id`에는 `:`(콜론)를 포함할 수 없다(키 구분자 충돌 방지). card_id 형식은 정규식 `^[a-z][a-z0-9-]{2,63}$`(소문자 시작, 소문자·숫자·하이픈, 3~64자, 대문자·언더스코어·콜론 금지)를 따르며 콜론을 허용하지 않는다(SoT _인터페이스계약 §1; 정의는 카드규격·생성규칙 SoT). card_id 유일성 범위 = deck namespace 내, 전역 유일성 = `namespace + ":" + card_id` 조합. localStorage 키 = `STORAGE_KEY_PREFIX + deckNamespace + ":" + card_id` 계열에서 콜론이 분리자이므로 양측 콜론 금지로 안전. `deckNamespace`는 콜론 미포함 문자열이어야 하며 위반 시 키 생성 단계에서 `ScoreInputError` 계열로 거부(한글 namespace는 콜론만 없으면 허용 — 키 충돌 없음).

### 4.2 스토어 구조
```ts
type MetaStore = {
  schema_version: number;      // = SCHEMA_VERSION, 마이그레이션 기준
  created_at: number;          // epoch ms
  last_opened_at: number;
}

type ProgressStore = {
  schema_version: number;      // 쓰기 시점 버전(키별 독립 기록)
  deck_namespace: string;
  cards: Record<string, CardProgress>;   // card_id → progress
}

type CardProgress = {
  card_id: string;             // 작성자 명시 안정 id(content-hash 아님, v4 §5)
  box: number;                 // BOX_MIN..BOX_MAX
  due_at: number;              // epoch ms, 다음 출제 가능 시각
  graduated: boolean;          // BOX_MAX에서 cold정답 도달 여부, 기본 false
  cold_attempts: number;       // 누적 cold 시도수(skip 포함), 기본 0
  cold_correct: number;        // 누적 cold 정답수, 기본 0
  last_attempt_at: number | null;  // 마지막 시도 시각, 기본 null
  last_verdict: Verdict | null;    // 기본 null ("correct"|"incorrect"|"skip")
}
```
- **card_id 안정성 계약(v4 §5)**: card_id는 작성자가 부여한 안정 식별자. 콘텐츠 오탈자 수정으로 **변하면 안 됨**(오답이력 보존). 생성규칙이 보장, 엔진은 키로만 사용.
- 신규 카드 첫 저장 시 CardProgress 기본값: `{box: BOX_MIN, due_at: 0, graduated: false, cold_attempts: 0, cold_correct: 0, last_attempt_at: null, last_verdict: null}`. (`due_at: 0` = 즉시 due)

### 4.3 마이그레이션 shim 계약
```ts
// 읽기 시 항상 통과. 저장은 항상 최신 SCHEMA_VERSION로.
function loadProgress(deckNamespace: string): ProgressStore
// 내부:
//   raw = localStorage.get(key)
//   raw 없음 → 빈 ProgressStore(schema_version=SCHEMA_VERSION) 반환
//   parse 실패 → 손상 폴백(아래)
//   raw에 schema_version 필드 없음 → version 0(legacy)로 간주, migrate 체인 진입
//   raw.schema_version < SCHEMA_VERSION → migrate(raw) 체인 적용
//   raw.schema_version > SCHEMA_VERSION → throw SchemaVersionError(다운그레이드 금지)
//   raw.schema_version == SCHEMA_VERSION → 그대로

function migrate(store: any): ProgressStore   // v(n) → v(n+1) 단계 체인
function saveProgress(store: ProgressStore): void  // schema_version=SCHEMA_VERSION로 강제
```
- **마이그레이션 규칙**: 버전별 변환 함수를 단계적으로 체인(`v0→v1→v2...`). 각 단계는 순수함수, 누락 필드는 §4.2 기본값으로 채움(데이터 손실 금지).
- **버전 필드 부재**: `schema_version` 키가 없는 JSON은 version `0`(레거시)으로 분류하여 migrate 체인의 시작점으로 본다(손상으로 보지 않음 — 객체 parse 자체는 성공한 경우).
- **손상/복구불가 폴백**: JSON `parse` 실패 → 빈 ProgressStore로 폴백 + 콘솔 경고 + 원본 문자열을 `STORAGE_KEY_PREFIX + deckNamespace + ":progress.bak"`에 보존(데이터 보호). throw 하지 않음(앱 무중단).
- **migrate 중 예외**: migrate 단계 함수가 throw하면 parse 실패와 동일 처리(원본 `.bak` 보존 + 빈 스토어 폴백 + 경고). 마이그레이션 실패는 "복구 가능 손상"으로 분류(진행불가가 아님).
- `migrate`는 현재 v0→v1 항등에 가까우나, **shim 구조·schema_version 필드는 v1부터 필수**(미래 마이그레이션 자리 확보).

### 4.4 쿼터/용량·동시쓰기
- localStorage 5MB 한계는 known tradeoff(v4 §8). 저장 실패(`QuotaExceededError`) 시 → throw `StorageQuotaError`, UI에 export 백업 권고 신호 전달. 엔진은 자동 삭제/축출 안 함.
- **멀티탭 경합**: 동일 `deckNamespace`를 여러 탭에서 동시 저장하면 last-write-wins다(v1은 단순 저장). 탭 간 동기화(`storage` 이벤트 구독)는 v1 비범위 — known limitation으로 둔다.

## 5. deck 네임스페이스 + manifest 로딩

### 5.1 로딩 방식 (file:// fetch 회피, v4 §5)
- deck/manifest = generate.py가 **전역 script 등록 JS**로 출력(`fetch` 금지 — file:// 차단 회피).
- 각 deck 파일은 로드 시 전역 레지스트리에 자기 등록(전역 변수·키 구체형은 `생성규칙.md` §2가 계약; 엔진은 등록된 레지스트리를 읽기만 한다).
- manifest = 사용 가능 deck 레지스트리(전역). 엔진은 전역 manifest·deck 레지스트리에서 읽으며 네트워크 호출 0.

### 5.2 manifest 엔트리(엔진이 읽는 형태)
```ts
type DeckManifestEntry = {
  namespace: string;           // deck 식별자(레지스트리 키)
  title: string;               // 표시명
  card_count: number;          // 카드 수(빌드 시점)
  config_var?: string;         // 과목 config 전역 경로(정규화·동의어·채점매핑)
  built_at: number;            // 빌드 epoch ms
}
```
- 전역 변수명·키 네임스페이스(`window.__DECKS__` 등)의 정확한 형태는 생성규칙 §2 소관. 엔진은 manifest가 가리키는 레지스트리를 통해 deck에 접근한다.

### 5.3 DeckData / CardDef
```ts
type DeckData = {
  namespace: string;
  cards: CardDef[];
}
type CardDef = {
  card_id: string;             // 안정 id(§4.2). §1 정규식(`id` 금지)
  schema_version: number;      // 마이그레이션용 카드 스키마 버전. 현재 1(SoT §14). generate.py 산출
  subject: string;             // deck namespace 키(필수, SoT §14 §2)
  unit: string;                // 단원 id(필수, SoT §14)
  type: "func" | "proc" | "recall_seq" | "cloze" | "judge";  // 카드 5유형 enum(SoT §3, 풀네임 function/procedure/sequence/judgment 폐기). 토큰 정의 SoT = 카드규격.md/생성규칙.md
  grade_mode: ScoreMode;       // §3.1. 필드명 = grade_mode (SoT §14; score_mode 폐기)
  front: object;               // 앞면(type별 키: prompt/text/scenario/options)
  back: object;                // 뒷면({detail?, note?})
  answer_spec: AnswerSpec | null;  // §3.3. 산출원 = generate.py(SoT §13); self면 null(non-nullable 선언 금지). 엔진은 answer_spec만 소비(평면 키 미수용)
  tags: {                      // 태그 오브젝트(최상위 직렬화 금지, SoT §14)
    weight: number;            // 빈출·배점 가중 [1,10] 기본 5 (§2.3, SoT §6)
    area?: "written" | "practical";   // 영역 enum (SoT §11)
    subarea?: "computer" | "spreadsheet" | "database" | "excel" | "access";  // 세부영역 enum (SoT §11)
  };
  links: {                     // 링크 오브젝트(최상위 직렬화 금지, SoT §14)
    concept_ref?: string;      // 개념서 섹션 id(형식은 개념서규격.md 소관, SoT §12). 개념서 미동반 시 보류(W-REFDEFER)
  };
  enabled: boolean;            // 기본 true(필수, SoT §14)
}
```
- 카드 5유형 ↔ grade_mode 매핑(SoT §4, 카드규격 §2.6 권위): `func`→`exact`(또는 self), `proc`→`keyword`(또는 self), `recall_seq`→`exact`(=순서포함 배열 완전일치, §3.3·SoT §9, 또는 self), `cloze`→`cloze`(또는 self), `judge`→`exact`(또는 self). 구 매핑 sequence→keyword·judgment→keyword는 **폐기**. 매핑 검증은 **카드규격.md/생성규칙.md 소관**(빌드타임). 엔진은 `grade_mode ∈ ScoreMode` 열거값만 신뢰·강제하며, 알 수 없는 grade_mode 값은 `ScoreInputError`.
- **중첩 구조 강제(SoT §14)**: `weight`/`area`/`subarea`는 `tags` 오브젝트 안, `concept_ref`는 `links` 오브젝트 안. 최상위 직렬화 금지. `related_cards`(concept→card 방향)는 개념서 ConceptSection 필드이지 카드 `links` 필드가 아님.
- **필수 필드(SoT §14)**: `subject`·`unit`·`enabled` 누락 금지. `answer_spec`은 `AnswerSpec | null`(self면 null, non-nullable 선언 금지).

### 5.4 로딩 함수
```ts
function getManifest(): { schema_version:number, decks:DeckManifestEntry[] }
```

## 6. 에러 규약

| 에러 클래스 | 발생 조건 | 처리 |
|---|---|---|
| `ScoreInputError` | mode↔userAnswer 타입 불일치, cloze 빈칸수 불일치, self 비허용값, 알 수 없는 grade_mode, 키에 콜론 포함 | throw, 호출자 catch |
| `SchemaVersionError` | 저장 버전 > 코드 버전(다운그레이드) | throw, 로드 중단(데이터 보호) |
| `StorageQuotaError` | localStorage 쓰기 QuotaExceeded | throw, export 권고 신호 |
| `ManifestMissingError` | 전역 manifest 부재 | throw, 앱 부트 실패 |
| `DeckNotFoundError` | 요청 namespace deck 미등록 | throw |

- 에러는 식별 가능한 분류(고유 `name`/코드)를 가질 것 — 구현은 `Error` 확장 또는 동등한 식별 가능 형태면 된다.
- **복구 가능 손상**(JSON parse 실패·migrate 예외)은 throw가 아니라 폴백+백업(§4.3) — throw는 "진행 불가"에만.

## 7. 대시보드 집계 (getDashboardData)

> 구현 소유 = 엔진. UI규격 §8 `getDashboardData()` 반환 형상과 **1:1 일치** 필수.
> 입력 = 현 localStorage progress(`ProgressStore`) + deck(`DeckData`). **순수 집계, 부수효과 없음**.

### 7.1 함수 시그니처

```ts
// 순수 집계 함수. 부수효과 없음. now는 호출자가 주입(테스트 결정성).
// deck: 활성 namespace의 DeckData(§5.3). progress: 현 ProgressStore(§4.2).
function getDashboardData(
  deck: DeckData,
  progress: ProgressStore,
  now: number             // epoch ms
): DashboardData

type DashboardData = {
  by_area:     ByAreaEntry[];      // (area,subarea) 쌍별 인출성취율. SoT §15
  weakness:    WeaknessEntry[];    // (area,subarea,unit) 단원별 오답률, 내림차순 정렬. SoT §15
  pass_path:   PassPathEntry[];    // (area,subarea) 쌍별 합격경로 커버리지×숙달+미달위험. SoT §15
  completion:  CompletionEntry[];  // (area,subarea) 쌍별 Leitner box 분포(정복률 box3+). SoT §15
}
```

### 7.2 반환 타입 상세

```ts
// ① by_area — (area, subarea) 쌍별 인출성취율 (cold_correct / cold_attempts)
//   집계 단위 = {area, subarea} 쌍(최대 5개: written×3 + practical×2, SoT §11).
type ByAreaEntry = {
  area:           "written" | "practical";  // §11 area enum
  subarea:        "computer" | "spreadsheet" | "database" | "excel" | "access";  // §11 subarea enum
  retrieval_rate: number | null;            // cold_correct / cold_attempts ∈ [0,1].
                                            // null = 해당 (area,subarea) cold_attempts == 0(데이터 부족)
}

// ② weakness — (area, subarea, unit) 단원별 오답률 내림차순
type WeaknessEntry = {
  area:       "written" | "practical";  // §11 area enum
  subarea:    "computer" | "spreadsheet" | "database" | "excel" | "access";  // §11 subarea enum
  unit:       string;    // CardDef.unit
  wrong_rate: number;    // cold 오답 / cold_attempts ∈ [0,1]. cold_attempts == 0 단원 제외
}
// 정렬: wrong_rate DESC → unit ASC(결정성)

// ③ pass_path — (area, subarea) 쌍별 합격경로(커버리지 × 숙달, 미달위험 플래그)
//   집계 단위 = {area, subarea} 쌍(SoT §15).
//   target(int): written subarea = 40, practical subarea = 70 (합격 floor, 과목규격 상수).
//   coverage = 해당 (area,subarea) cold_attempts >= 1인 카드 수 / 전체 카드 수 ∈ [0,1].
//   mastery  = 해당 (area,subarea) cold_correct / cold_attempts >= 1인 카드 수 ∈ [0,1];
//              cold_attempts >= 1인 카드 0이면 0.
//   progress = coverage × mastery ∈ [0,1].
//   status 판정(계약, SoT §15):
//     r = progress, t = target / 100
//     r >= t           → 'safe'
//     r >= t * 0.7     → 'watch'
//     그 외             → 'danger'
//   UI는 status enum을 렌더만(임계 재계산 금지).
type PassPathEntry = {
  area:         "written" | "practical";
  subarea:      "computer" | "spreadsheet" | "database" | "excel" | "access";
  target:       number;    // 정수. written subarea=40, practical subarea=70 (SoT §15)
  coverage:     number;    // ∈ [0,1]
  mastery:      number;    // ∈ [0,1]
  progress:     number;    // = coverage × mastery ∈ [0,1]
  status:       "safe" | "watch" | "danger";  // 엔진 산출, UI 렌더만
}

// ④ completion — (area, subarea) 쌍별 Leitner box 분포(정복률 = box3+ 비율)
//   집계 단위 = {area, subarea} 쌍.
//   box_dist: 해당 (area,subarea) 카드 중 각 box에 속한 카드 수(progress 미존재 = box1 미학습으로 계산).
//   graduated 카드는 box3으로 집계(BOX_MAX=3, §0·§1.1).
//   mastery_rate = box3 카드 수 / 전체 카드 수 ∈ [0,1].
type CompletionEntry = {
  area:         "written" | "practical";
  subarea:      "computer" | "spreadsheet" | "database" | "excel" | "access";
  box_dist: {
    box1: number;   // int, box==1 카드 수(미학습 포함)
    box2: number;   // int, box==2 카드 수
    box3: number;   // int, box==3 카드 수(graduated 포함)
  };
  mastery_rate: number;   // box3 카드 수 / 전체 카드 수 ∈ [0,1]. "정복률 box3+"
}
```

### 7.3 집계 규칙 (계약)

- **입력 범위**: `enabled == true` 카드만 집계 대상. `enabled == false` 카드는 분자·분모 모두 제외.
- **progress 미존재 카드**: `ProgressStore.cards`에 card_id 항목이 없으면 미학습(`cold_attempts == 0`, `box == BOX_MIN`)으로 간주. 덱에서 삭제된 고아 progress는 집계에 불포함(cards 기준).
- **area/subarea 없는 카드**: `tags.area` 또는 `tags.subarea`가 없는 카드는 어느 ByAreaEntry·PassPathEntry·CompletionEntry에도 포함하지 않는다(weakness 집계에는 unit 기준으로 포함하되, area/subarea 필드는 해당 카드의 tags에서 그대로 전달; 두 필드 모두 없는 카드는 weakness에서도 제외).
- **부수효과 없음**: `getDashboardData`는 localStorage 읽기·쓰기 없음. progress·deck 객체를 변이하지 않음. 호출 전 `loadProgress()`·`loadDeck()`으로 준비한 값을 주입한다(읽기 책임은 호출측).
- **결정성**: 같은 `(deck, progress, now)` → 같은 `DashboardData`. 정렬·status 판정 모두 결정적(§7.2 정렬 키 명시).

### 7.4 테스트 게이트 (§8 연계)

다음 케이스를 §8 테스트셋에 추가한다:
- `by_area`: (area,subarea) 쌍별 집계, cold_attempts 0 → null, 정상 집계 비율 정밀도.
- `weakness`: 오답률 내림차순 정렬, cold_attempts 0 단원 제외, tie-break 결정성(unit ASC).
- `pass_path`: safe/watch/danger 임계(r≥t safe / r≥0.7t watch / else danger, t=target/100), written subarea=40·practical subarea=70 기본값, progress=coverage×mastery 정밀도.
- `completion`: (area,subarea) 쌍별 집계, progress 미존재 카드 → box1 계산, graduated → box3 집계, mastery_rate 정밀도.

## 8. 테스트셋 동반 필수 (v4 §5 ④검증 게이트)

본 계약의 다음 요소는 테스트셋 없이는 게이트 통과 불가:
- Leitner 전이표(§1.3) 전 행 — cold/warm × correct/incorrect 4조합 + cold/warm skip + box 경계(BOX_MIN↔BOX_MAX) + 졸업/졸업후재전이.
- 출제 우선순위(§2.2) — due-first, weight 정렬, tie-break 결정성, 빈 큐, 고아/신규 progress, limit 적용, D-day 전박스 소환·정렬.
- 채점 4모드(§3.1) — exact 복수정답/정규화, keyword 전부포함 이진, cloze 빈칸수·이진, self 패스스루, 빈 입력 incorrect.
- 정규화(§3.4) — 규칙 id 배열 순서 적용·각 카탈로그 규칙(nfkc/trim/collapse_space/strip_all_space/lower/fullwidth_to_halfwidth/unify_cell_dollar/unify_arg_sep/strip_trailing_paren)·동의어(lower 직후 치환).
- recall_seq 순서 채점(§3.3) — sequence 순서포함 원소단위 완전일치, 길이불일치 incorrect, 부분점수 없음.
- 마이그레이션 shim(§4.3) — 버전 없음(v0)/하위/동일/상위(throw)/손상JSON 폴백/migrate예외 케이스.
- 대시보드 집계(§7) — by_area null(cold_attempts=0)·(area,subarea) 쌍별 집계·정상 비율; weakness 내림차순·제외 케이스·tie-break(unit ASC); pass_path safe/watch/danger 임계(r≥t·r≥0.7t·else, t=target/100, written subarea=40·practical subarea=70)·progress=coverage×mastery; completion (area,subarea) 쌍별·progress 미존재→box1·graduated→box3·mastery_rate.

테스트는 `now` 주입으로 시간 결정성 보장(§0). 테스트 수단은 **무의존 stdlib 한도와 무관**하다(불변경계의 "무의존 stdlib"는 generate.py 런타임 산출에 한정 — 개발용 테스트 러너는 그 제약 밖). JS 코어 테스트 러너(브라우저 콘솔/Node 내장 test/순수함수 assert 등)는 ④검증 게이트에서 결정한다. [OPEN-A]

## 9. 불변 계약 요약 (v4 모순 금지 체크리스트)

1. score()는 정답률·합격확률·자동채점 결과를 **산출하지 않는다**. (verdict[correct/incorrect] + 피드백만)
2. 박스 이동은 **cold 첫시도 정답**에만 의존, warm은 박스무관.
3. 모든 채점은 **이진 정/오답**(부분점수·가점·부분승급 없음). 세션내 재큐가 교정장치.
4. 런타임 네트워크 호출 0(fetch 금지), 전역 script 등록 레지스트리 읽기만.
5. card_id = 작성자 안정 id, 콘텐츠 수정에 불변, content-hash 아님.
6. 스키마는 schema_version 필드 + 마이그레이션 shim를 v1부터 보유.
7. 과목 특화(정규화·동의어·채점매핑·Leitner 간격)는 config 주입, 엔진 코드 무오염(과목 어휘·간격 하드코딩 금지, 코어 기본값만).
8. 전역 synonyms 치환 위치 = 정규화 소문자화(lower) 직후·비교 직전(카드·생성·엔진 동일).
9. `getDashboardData()`는 **순수 집계**(부수효과·localStorage 쓰기 없음). 합격확률·예측값 산출 금지(v4 §2). 반환 최상위 키 `by_area`·`weakness`·`pass_path`·`completion`, 식별 필드 `area`+`subarea`(area_id 금지), SoT _인터페이스계약 §15와 1:1 일치.
