# _인터페이스 계약 (cross-contract SoT)

> buildflow ② 통합 산출물(opus 직접). 계약 간 공유 인터페이스의 **유일 진실원천(SoT)**.
> 다른 규격 md와 충돌 시 **이 문서가 우선**. 모든 규격은 아래 결정에 맞춰 정렬한다.
> 발생 경위: 병렬 마감서 5개 계약이 공유 토큰·타입을 제각각 정의 → 정합검사 ok:false(불일치 12 + dangling 3) → opus 통합 확정.

---

## 1. card_id
- 정규식(유일): `^[a-z][a-z0-9-]{2,63}$` — 소문자 시작, 소문자·숫자·하이픈, 3~64자. **대문자·언더스코어·콜론 금지**(대소문자 FS 충돌·localStorage 키 콜론 분리자 회피).
- 하이픈 **세그먼트 강제 안 함**(있어도 됨).
- **유일성 범위 = deck namespace 내**. 전역 유일성 = `namespace + ":" + card_id` 조합으로 성립(문자열 단독 전역유일 강제 안 함).
- 검사: 정규식 + deck 내 중복(E-IDDUP). 생성규칙의 전역 문자열 유일 검사 → **deck 범위로 변경**.

## 2. deck namespace
- **card_id는 namespace를 인코딩하지 않는다.** deck/namespace는 카드의 `subject` 필드 + manifest에서 온다.
- "card_id에서 deck 추출" 규칙은 **존재하지 않음(설계상 불가)** — 개념서 등에서 이 추출을 기대하지 말 것. deck = `subject` 필드로 직접 참조.
- localStorage 키 = `namespace + ":" + card_id` (콜론 = 분리자, 양측에 콜론 금지라 안전).

## 3. card type enum (유일 토큰)
- `func | proc | recall_seq | cloze | judge` — **이 5개 토큰만**. (풀네임 `function/procedure/sequence/judgment` 폐기.)
- 생성규칙 CardDef·엔진 CardDef 전부 이 토큰으로 정렬.
- 개념서의 *섹션 ref_type*(`function|procedure|concept`)은 **별개 어휘**(카드 type 아님) — 혼동 방지 위해 개념서는 "concept section type"으로 명명, 카드 type과 분리.

## 4. type × grade_mode 매트릭스 (카드규격 §2.6 권위)
| type | 정답 grade_mode |
|---|---|
| `func` | `exact` (또는 `self`) |
| `proc` | `keyword` (또는 `self`) |
| `recall_seq` | `exact` = **순서포함 배열 완전일치** (또는 `self`) |
| `cloze` | `cloze` (또는 `self`) |
| `judge` | `exact` (또는 `self`) |
- 생성규칙의 sequence→keyword, judgment→keyword 매핑 **폐기**. recall_seq·judge = exact.
- → **엔진은 순서재현(ordered array) 채점을 반드시 구현**(§9 참조).

## 5. grade_mode / verdict 어휘 (유일 토큰)
- grade_mode: `exact | keyword | cloze | self`.
- verdict(채점결과·상태): **`correct | incorrect | skip`** — 모든 계약·localStorage·백업 동일. (`wrong` 폐기.)
- **self 입력**: 사용자 UI 입력은 `o|x`(UI/카드 레이어) → **앱(UI) 레이어가 `correct|incorrect`로 변환** 후 엔진 score()에 전달. 엔진 score()는 `correct|incorrect`만 수용(o/x throw). 변환 주체 = UI.

## 6. weight (빈출·배점)
- 타입·범위: `float`, **[1, 10]**. 미지정 시 **기본 5**.
- **freq 파생식 폐기 + freq 필드 자체 폐기**(dangling 제거). 카드규격 §7의 `freq` 필드 삭제 — 빈출·배점 신호는 `weight` 하나로 일원화. 작성자가 빈출 카드에 높은 weight 직접 부여. 미지정=5.
- 엔진: 범위 밖 입력 클램프, 비숫자 폴백 5(런타임 무중단). 생성규칙 산출도 [1,10].

## 7. 정규화(normalize) 모델 (카드규격 §3.5 권위)
- 모델 = **순서 있는 규칙 id 배열** `normalize: string[]` + 공유 카탈로그. 배열 **순서대로** 적용(= 계약).
- boolean 플래그 객체(엔진 NormalizeConfig / 생성 NormalizeOptions) **폐기** → 전부 id 배열 모델로.
- 공유 카탈로그(id → 동작): `nfkc · trim · collapse_space · strip_all_space · lower · fullwidth_to_halfwidth · unify_cell_dollar · unify_arg_sep · strip_trailing_paren` (정의는 카드규격 §3.5 카탈로그가 SoT).
- 카드 `grading.normalize`(id 리스트)를 엔진이 그대로 수용. 미지정 시 과목 config의 type별 기본 프로파일(id 리스트).

## 8. synonyms 사전
- **작성 형식(config)** = `{ 대표어: [동의어, ...] }` (사람 친화).
- **런타임 형식(엔진)** = `Record<동의어, 대표어>` (역인덱스).
- **변환 주체 = generate.py** (config의 대표어→배열을 역인덱스로 컴파일해 deck/엔진에 공급). 엔진은 역인덱스만 소비.
- 치환 위치 = 정규화 `lower` 직후·비교 직전(3계약 합의됨). 단위 = 토큰(공백분리) 정확일치. 다어절 치환 = v1 밖.

## 9. recall_seq 순서 채점 (엔진 필수 구현)
- 카드 `answer` = `string[]`(순서 의미). 엔진 AnswerSpec에 **순서배열 필드 추가**.
- exact 채점(recall_seq) = step별 정규화 후 **순서포함 원소단위 완전일치**. 부분점수 없음(이진).

## 10. cloze 빈칸 인덱스 = 0-base (통일)
- 마커 `{{0}}`, `{{1}}` …, `answer` 키 `"0"`,`"1"` … — **전부 0-base**. (카드규격의 1-base → 0-base로 변경.)
- 엔진 matched/missed, UI data-blank-index, 생성 idx 모두 0-base와 일치(off-by-one 변환 불필요).

## 11. area / subarea (2필드 통일)
- `area` enum: **`written | practical`** (2값).
- `subarea` enum: `computer | spreadsheet | database`(written) · `excel | access`(practical).
- 점결합 표기(`written.computer`) **폐기** → 항상 두 필드 {area, subarea}.
- 개념서·생성 config·UI 롤업 전부 {area, subarea} 2필드로 정렬.

## 12. 개념서 ↔ 카드 링크 (필드명 통일)
- 카드 → 개념: `links.concept_ref` (개념서 섹션 id).
- 개념 → 카드: `related_cards` (card_id[]).
- 엔진 CardDef의 `section_ref` → **`concept_ref`로 개명**(카드규격·개념서와 일치).
- 개념서 미동반 시 카드 `links.concept_ref` = **보류(deferred warning)** — concept_ref가 아직 빌드 안 된 개념서 섹션을 가리키면 *에러 아닌 경고*. **경고 코드명은 각 계약 자체 정의**(SoT는 의미만 강제, 코드명 통일 강제 안 함 — over-spec 회피). `related_cards` dangling(가리킨 card_id 부재) 검증 = **개념서 ConceptSection 소관** — 카드측 E-REF 아님(폐기, 카드규격서 제거).

## 13. answer_spec — 채점 데이터 조립 (generate.py 소유)
- 카드 작성 소스(콘텐츠 md / authoring) = 평면 키(`answer`/`grading.accept`/`grading.keywords`/`cloze answer` 등 카드규격 형태).
- **generate.py가 빌드시 평면 키 → `answer_spec` 객체로 조립**해 deck Card에 넣는다. **엔진은 `answer_spec`만 소비**(런타임 필드 분기 없음). 평면 키는 엔진에 노출 안 됨.
- type별 매핑(generate.py 소유):
  - `func` exact / `judge` exact → `answer_spec.accepted: string[]` (answer + grading.accept, OR 집합)
  - `proc` keyword → `answer_spec.requiredKeywords` (grading.keywords)
  - `recall_seq` exact → `answer_spec.sequence: string[]` (answer 배열, 순서 의미)
  - `cloze` → `answer_spec.blanks: string[][]` — `blanks[i]` = i번 빈칸(0-base 위치=인덱스, 마커 0..N-1 연속이라 위치=인덱스 안전)의 정답후보 배열(answer[i] + accept_map[i]). 별도 idx 필드 없음(위치가 인덱스).
  - `self` → `answer_spec` 생략/빈값(verdict는 UI 제공)
- dangling 해소: 엔진 CardDef.`answer_spec`의 산출원 = generate.py(본 §13). **카드규격은 answer_spec을 "생성물(작성 대상 아님)"으로 명시**, 평면 키가 authoring SoT.

## 14. deck Card 직렬화 스키마 (generate.py 산출 → 엔진 소비, 유일 필드명)
generate.py가 deck.js에 내보내는 Card 객체의 **정확한 필드명·타입**. 모든 계약(생성 산출·엔진 소비·UI)이 이 이름을 쓴다. (id→card_id, grade/score_mode→grade_mode, blanks 형상 충돌 종결.)

| 필드 | 타입 | 비고 |
|---|---|---|
| `card_id` | string | §1 정규식. (`id` 금지) |
| `schema_version` | int | 마이그레이션용. 현재 `1`(int, `"1.0"` 문자열 금지). 카드·생성·엔진 deck Card + 개념서 ConceptSection 전부 int 통일 |
| `subject` | string | deck namespace 키 |
| `unit` | string | 단원 id |
| `type` | enum `func\|proc\|recall_seq\|cloze\|judge` | §3 |
| `grade_mode` | enum `exact\|keyword\|cloze\|self` | §5. **유일 이름 = `grade_mode`** (`score_mode`·`grade` 폐기) |
| `front` | object | type별 키(prompt/text/scenario/options) |
| `back` | object | {detail?, note?} |
| `answer_spec` | object\|null | §13. self면 null |
| `answer_spec.accepted` | string[]? | func/judge exact |
| `answer_spec.requiredKeywords` | string[][]? | proc keyword. 내부배열=한 필수그룹의 동의어 후보(any-of), 모든 그룹 필수(v1) |
| `answer_spec.sequence` | string[]? | recall_seq, 순서 의미 |
| `answer_spec.blanks` | string[][]? | cloze, §13(위치=0-base 인덱스) |
| `answer_spec.normalize` | string[] | 이 카드에 적용할 정규화 규칙 id 순서배열(§7) |
| `tags` | object | {`weight`: float[1,10] 기본5, `area`: enum, `subarea`: enum} (§6,§11) |
| `links` | object | {`concept_ref`?: string} (§12) |
| `enabled` | bool | 기본 true |

- synonyms 사전은 **카드 필드 아님** — deck/config 레벨(generate.py 컴파일 역인덱스, §8). 엔진 score()가 deck 레벨에서 받음.
- **중첩 구조 강제**: `weight`/`area`/`subarea`는 `tags` 오브젝트 안, `concept_ref`는 `links` 오브젝트 안(최상위 직렬화 금지). 카드 authoring 구조와 동일(authoring=emitted, 변환 최소).
- `links`는 카드측 `{concept_ref?}` **만**. `related_cards`(concept→card 방향)는 **개념서 ConceptSection 필드**이지 카드 필드 아님(§12). 카드 `links`에 related_cards 두지 말 것.
- **필수 필드 누락 금지**: `subject`·`unit`·`enabled`는 deck Card 필수. 엔진 CardDef·생성 Card 산출 스키마 모두 포함해야 함. `answer_spec`은 `object|null`(self면 null, non-nullable 선언 금지).
- **authoring 공급 경로**: `subject`·`unit`은 카드 authoring(front-matter 또는 카드 메타)에서 작성자 공급 → generate.py가 deck Card로 통과(생성규칙 authoring 스키마에 `unit` 키 포함 필수). `enabled` 미작성 시 generate.py 기본 true 주입. `schema_version`·`answer_spec`·`tags`/`links` 구조 = generate.py 빌드 산출(작성 대상 아님).
- **`answer_spec.normalize`는 answer_spec 내부 필드**(ScoreInput 최상위 아님). 엔진 AnswerSpec 정의에 포함.
- 엔진 CardDef·생성 Card 산출 스키마·UI getNextCard 반환 = 위 필드명과 1:1.

## 15. getDashboardData() 반환 계약 (엔진 구현, UI 소비)
- 반환 = object, 최상위 키 **4개**: `by_area` · `weakness` · `pass_path` · `completion`. UI는 이 키명으로 읽는다(`progress`·단일 `retrieval_rate` 등 다른 키 금지).
- 공통 식별 필드명 = **`area`(enum written|practical) + `subarea`(enum)** — `area_id` 금지.
- `by_area: AreaStat[]` = `{area, subarea, retrieval_rate: float01}` (영역별 인출성취율 proxy)
- `weakness: WeaknessEntry[]` = `{area, subarea, unit: string, wrong_rate: float01}` (오답률 내림차순)
- `pass_path: PassPathEntry[]` = `{area, subarea, target: int, coverage: float01, mastery: float01, progress: float01(=coverage×mastery), status: enum 'safe'|'watch'|'danger'}`.
  - **status는 엔진이 산출**(target 상대 기준: r=progress, t=target/100 → r≥t `safe` / r≥0.7t `watch` / else `danger`). **UI는 status enum을 렌더만, 임계 재계산 금지**(UI 절대임계 0.75/0.50 폐기).
  - target = 필기 subarea **40** · 실기 subarea **70**(과목별 합격 floor). 필기 평균60 교차검사는 엔진 별도 집계(본 계약 밖).
- `completion: CompletionEntry[]` = `{area, subarea, box_dist: {box1:int, box2:int, box3:int}, mastery_rate: float01(=box3/total)}` (정복률)
- 순수 집계(부수효과 없음). 입력 = 현 localStorage progress + deck. 엔진 §7 = 이 계약, UI §8 = 이 계약 소비, **1:1**.

## 16. 런타임 와이어링 (generate.py 산출 ↔ app 소비, 유일 이름)
- **전역 등록(유일)**: `window.DECKS[deck_id] = {meta, cards}` · `window.MANIFEST[subject] = {...}` · `window.SYNONYMS[subject] = {동의어: 대표어}`. 이중밑줄(`__DECKS__` 등) **폐기**. `window.DECKS`=**deck_id 키**, manifest·synonyms=**subject 키**.
- **`MANIFEST[subject].decks` = 객체 배열** `[{deck_id, title, area, subarea, card_count, src}]`(사이드바 네비 메타 필요 — id-only 아님). app.js는 각 항목 `.deck_id`로 `DECKS[deck_id]` 소비(문자열 배열도 하위호환 허용).
- **config 포맷 = JSON** (`과목/<과목폴더>/config/규칙.json`). `.config.js` **폐기**. generate.py가 stdlib json으로 읽음.
- **subject id = `comp1`**(컴활1급). 폴더명 `컴활1급`과 별개.
- **부트 진입점 = `window.APP.init()`**. index.html이 [생성물 scripts(synonyms→manifest→decks) → app.js] 로드 후 `window.APP.init()` 호출. app.js는 `window.APP = { init, ... }` 노출. (`__APP_INIT__` 폐기.)
- **콘텐츠 경로**: `과목/컴활1급/콘텐츠/**/*.md`(1단원 = `필기_스프레드시트/01_단원.md`). 산출 = `과목/컴활1급/생성물/`(decks/*.js + manifest.js + synonyms.js).
- §14 준수: answer_spec에 `normalize: string[]` 포함(카드 grading.normalize, 미지정 시 config type별 기본 프로파일).
- **UI 의존성 = CDN 허용**(사용자 결정, 확장 유연성 우선). app/는 인터넷 CDN 자유 사용(Google Fonts·KaTeX·marked 등 `<link>`/`<script src>`). **단**: ① 카드/진도 데이터는 여전히 `window.DECKS/MANIFEST/SYNONYMS` 전역(로컬 파일 fetch 금지 — file:// CORS) ② 앱은 정적 HTML 더블클릭 유지(폰트·라이브러리만 원격, 인터넷 필요). generate.py는 그대로 stdlib 무의존(빌드 레이어 별개).

## 적용 지시 (전파 대상)
각 규격은 위 12항에 맞춰 자기 문서의 충돌 부분을 수정한다. 충돌 시 본 문서 우선. 본 문서에 없는 세부는 각 규격 자체 정의 유지.
