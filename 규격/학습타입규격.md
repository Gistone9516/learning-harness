# v5 학습타입 규격 (능력 레지스트리)

> buildflow ② per-folder 계약. **SoT(`_인터페이스계약.md`)에 conform**, 특히 §4 `LearningCapability`.
> 범위: 4계층 학습 능력을 **레지스트리 항목**으로 정의(`capability_id`가 권위). 봇계약이 `handler`를 구현 매핑.
> 출처 = `_이데이션_능력카탈로그.md`(+ `_이데이션_원본.json`). tier: core=일상 핵심, extension=opt-in.
> 표 칼럼: `capability_id` | tier | primitives(harness) | engine_fns / needs_ai | 상호작용 계약(handler 책임).

## 활성화·기본값 (주입인터페이스 §7 연동)

- config `capabilities.enabled` 미지정 → **core tier 전체 활성**(layer 4 gating·event·heartbeat·coalesce 포함).
- extension·layer3(ai)는 명시적 opt-in. 알 수 없는 `capability_id` = `ContentInjectionError`.
- 능력 간 의존(예: `ai_*`는 `gating`·`_invoke`; `srs_due_alert`는 `queue`·`persist`)은 봇계약이 부트 시 검증(누락 의존=경고+비활성).
- **needs_ai(SoT §4 필수)**: layer 1·2·4 = `False`, layer 3 = `True`(전부 행에 확정값). 위 표의 layer로 결정.
- **사이드카(SoT §2.1)**: CardProgress 밖 보조 상태를 쓰는 능력 — `confidence_rate`(confidence) · `read_resume`(read_pos) · `elaborate_ask`(elaboration) · `srs_due_alert`(alert_sent 중복방지) · adaptive_weight(weight 재산출) — 은 SoT §2.1 사이드카 계약을 따른다(봇/harness 관리, 엔진 스키마 무오염, 경로 `<mount>/_상태/sidecar-<capability_id>-<deck>.json`).

> [OPEN-B] 해소(봇계약): `handler`는 `capability_id → 봇 핸들러 함수` 디스패치 테이블로 매핑.
